## 一、历史演进：从"氛围编程"的宿醉到规范驱动的崛起

### 一切从 Karpathy 的一条推文开始

2025 年 2 月，OpenAI 联合创始人 Andrej Karpathy 在社交媒体上发了一段话，大意是：他现在写代码的方式变了，不再认真看代码，而是把想法告诉 AI，AI 生成什么就用什么，感觉不对就重新生成，整个过程靠"感觉"（Vibe）驱动。他把这种方式叫做 **Vibe Coding**。

这条推文在开发者社区引爆了。对于很多人来说，这不是什么新鲜事——他们早就在这么干了，只是没有一个好听的名字。Karpathy 的命名给了这种行为一种合法性：你不是在偷懒，你是在 Vibe Coding。

接下来的几个月，Vibe Coding 成了 2025 年 AI 编程圈最热的词。Cursor、Windsurf、GitHub Copilot 的用户量暴涨，"10 倍速开发"的故事在各个技术社区流传。非程序员也开始用 AI 写代码，产品经理、设计师、甚至完全没有编程背景的创业者，都开始"Vibe"出自己的 MVP。

但蜜月期没有持续太久。

### 宿醉来得很快

大约从 2025 年中开始，一批反思性的文章和数据开始出现。

Veracode 的 2025 年调查发现，AI 生成的代码中 **45% 包含安全漏洞**。CodeRabbit 对 470 个开源 Pull Request 的分析显示，AI 协作编写的代码比纯人类编写的代码"主要问题"多 1.7 倍，配置错误多 75%，安全漏洞多 2.74 倍。更具讽刺意味的是，一项研究表明，**熟练的开发人员在使用 AI 编码工具时，生产力反而下降了 19%**——尽管他们自己预测会提高 24%。

GitClear 对 2.11 亿行代码的研究则揭示了另一个维度的问题：AI 工具普及后，代码复制粘贴率上升了 48%，而真正的重构活动下降了 60%。代码在变多，但质量在变差，可维护性在下降。

这种现象被称为"Vibe Coding 宿醉"。它的根本原因不难理解：Vibe Coding 的核心假设是"AI 能读懂你的意图"，但 LLM 本质上是模式补全机器，它擅长生成"看起来像那么回事"的代码，而不是"真正符合你的系统约束、安全要求和架构决策"的代码。

### SDD 的登场：一个旧理念的新包装

就在 Vibe Coding 的问题开始暴露的同时，一个新的词开始在工程圈流传：**Spec-Driven Development（SDD，规范驱动开发）**。

SDD 的核心思想其实并不新鲜。它脱胎于软件工程中由来已久的"契约优先"（Contract-First）思想，以及 2000 年代流行的行为驱动开发（BDD，Behavior-Driven Development）。BDD 的核心理念是：在写代码之前，先用自然语言描述系统的行为（Given/When/Then 格式），让这些描述成为开发、测试和业务之间的共同语言。

SDD 把这个理念搬到了 AI 时代，并做了一个关键的重新定位：**规范（Spec）不只是写给人看的文档，它是写给 AI 看的"执行指令"**。

2025 年下半年，Thoughtworks 在其技术雷达报告中将 SDD 列为值得关注的重要实践。技术总监 Liu Shangqi 的一句话被广泛引用："Spec-driven development may not have the visibility of a term like vibe coding, but it's nevertheless one of the most important practices to emerge in 2025."（规范驱动开发也许没有 Vibe Coding 那么高的曝光度，但它是 2025 年涌现出的最重要的实践之一。）

### 工具生态的快速成型

2025 年到 2026 年初，围绕 SDD 的工具生态快速成型，几个主要玩家相继登场：

**GitHub Spec Kit**（2025 年中）：GitHub 官方推出的 SDD 工具包，基于 Markdown 文件，通过结构化的 spec 文件引导 AI 生成代码。它的设计哲学是"严格阶段门"——每个阶段必须完成才能进入下一阶段，强调可追溯性和团队协作。

**Amazon Kiro**（2025 年下半年）：AWS 推出的专属 IDE，深度集成了 SDD 工作流。Kiro 的特点是把 spec 文件直接嵌入 IDE 界面，开发者在写代码的同时就能看到对应的规范。但它的问题也很明显：锁定在 Kiro IDE 和 Claude 模型，不支持其他工具链。

**OpenSpec**（2025 年底，Fission-AI 开源）：定位为"通用开源框架"，支持 20+ 主流 AI 编程助手（Cursor、Claude Code、Windsurf、Copilot 等）。它的核心工作流是三条命令：`/opsx:propose`（提出变更，生成 proposal/specs/design/tasks 四类文档）、`/opsx:apply`（逐任务执行）、`/opsx:archive`（归档变更）。

**Tessl**（2026 年初）：走得更激进，直接把 spec 作为唯一维护的产物，代码完全由 spec 生成，不需要人工维护代码库。这是 SDD 理念的极端形态，目前还处于早期阶段。

2026 年 2 月，arxiv 上出现了一篇题为《Spec-Driven Development: From Code to Contract in the Age of AI Coding》的论文，系统性地梳理了 SDD 的理论基础和实践路径，标志着这个领域开始获得学术界的正式关注。

到 2026 年，SDD 已经从一个工程圈的小众讨论，演变成了 AI 编程领域的主流范式之一。阿里、腾讯等大厂的技术团队开始发布内部 SDD 实践经验，"规范驱动"成为 AI 原生研发的标配词汇。

## 二、SDD 的核心价值：它确实解决了一些真实问题

在深入讨论 SDD 的现存问题之前，有必要先承认它解决了什么——否则就无法理解为什么它会在短时间内获得如此广泛的认可。

**上下文锚定（Context Anchoring）**：Vibe Coding 最大的工程问题之一是"上下文腐烂"——随着对话进行，AI 会逐渐忘记早期的设计决策，代码质量直线下降。SDD 通过把所有决策固化在 Markdown 文件中，并纳入版本控制，解决了这个问题。任何时候，把 spec 文件交给 AI，它就能重建完整的上下文。

**幻觉抑制**：结构化的 spec 文件为 AI 提供了精确的输入，把 AI 的推理路径限制在一个更小的范围内，从而减少了"自由发挥"导致的幻觉。OpenSpec 的"小步快跑"模式（每次只处理一个原子任务）进一步强化了这一效果。

**审查范式转变**：Vibe Coding 时代，面对 AI 生成的上万行代码，人类根本无法逐行 Review。SDD 把"代码审查"变成了"规范审查"——在代码生成之前，先审查 spec 是否正确，成本更低，介入更早。

**知识持久化**：在 Vibe Coding 中，项目知识存在于聊天历史中，极易丢失。SDD 把知识沉淀在 `specs/` 目录中，即便更换 AI 模型，新模型也能通过阅读规范快速理解项目全貌。

这些价值是真实的，也是 SDD 能够快速获得认可的原因。但认可一个范式的价值，和承认它已经成熟、问题已经解决，是两回事。

## 三、现存的五大核心问题

SDD 在 2025-2026 年的快速崛起，掩盖了一个事实：这个范式本身还有大量未解决的根本性问题。Thoughtworks 的 Liu Shangqi 在肯定 SDD 价值的同时，也直接点出了这一点："Spec drift and hallucination are inherently difficult to avoid. We still need highly deterministic CI/CD practices to ensure software quality and safeguard our architectures."

以下是目前 SDD 实践中最核心的五个未解决问题。

### 问题一：Spec Drift——规范与代码的同步是个伪命题

SDD 的核心假设是"规范是单一事实来源（Single Source of Truth）"。但在实际工程中，这个假设很快就会被打破。

代码在变，规范却很难同步更新。一个典型的场景是：开发者按照 spec 生成了代码，然后在调试过程中直接修改了代码（因为这比修改 spec 再重新生成快得多），但没有回头更新 spec。几周后，spec 和代码之间出现了偏差。这种偏差被称为 **Spec Drift（规范漂移）**。

Spec Drift 的危险在于它是隐性的。代码能跑，测试能过，但规范已经不再准确描述系统的实际行为。当下一个开发者（或下一个 AI 会话）拿着过时的 spec 来生成新代码时，就会产生基于错误前提的实现。

更深层的问题是：**谁来维护规范的准确性？** 在传统软件工程中，文档腐烂（Documentation Rot）是一个老问题，解决方案是"代码即文档"（Code as Documentation）——让代码本身成为最准确的描述。SDD 试图反转这个逻辑，让规范成为最准确的描述，但它没有解决"谁来保证规范和代码同步"的问题。

目前的工具（OpenSpec、Spec Kit 等）都没有提供自动检测 Spec Drift 的机制。OpenSpec 的 `/opsx:archive` 命令可以归档变更，但它依赖开发者主动执行，而不是自动触发。这是一个依赖人类自律的解决方案，在工程实践中往往不可靠。

### 问题二：Spec 质量的冷启动困境——谁来写第一份好规范？

SDD 的工作流通常是这样的：人提出需求 → AI 生成 spec → 人审查 spec → AI 按 spec 生成代码。这个流程的前提是"人能够审查 spec 的质量"。

但这里有一个隐藏的循环依赖：**要审查一份好的 spec，你需要对系统有深入的理解；但如果你对系统有深入的理解，你可能根本不需要 AI 来帮你写 spec。**

对于有经验的开发者来说，SDD 的价值是显而易见的——他们能够快速识别 AI 生成的 spec 中的问题，补充边界条件，修正错误假设。但对于经验不足的开发者（或者非技术背景的用户，他们恰恰是 Vibe Coding 的主要受益者），他们往往无法判断一份 spec 是否足够好。他们会批准一份看起来合理但实际上遗漏了关键约束的 spec，然后得到一份看起来正确但实际上有严重缺陷的代码。

这个问题在多代理系统中更加严重。当多个 AI Agent 协作时，一个 Agent 生成的 spec 会被另一个 Agent 直接使用，没有人类在中间审查。如果第一个 Agent 的 spec 有问题，错误会被放大和传播，形成"幻觉循环"——Agent A 的错误输出被 Agent B 误判为正确，相互强化，最终产生一个看起来自洽但实际上完全错误的系统。

目前没有任何工具提供"spec 质量评估"的自动化机制。如何判断一份 spec 是否足够完整、足够准确、足够无歧义，仍然完全依赖人类的判断。

### 问题三：非确定性的验证缺口——规范说了，AI 不一定做到

即使有了高质量的 spec，SDD 还面临一个根本性的技术问题：**LLM 是非确定性的**。同样的 spec，不同的模型、不同的时间、不同的温度参数，可能生成完全不同的代码。

这意味着"spec 通过了审查"并不等于"生成的代码符合 spec"。你需要一个独立的验证机制来确认代码确实实现了 spec 中描述的行为。

传统的解决方案是测试——写测试用例来验证代码行为。但在 SDD 的语境下，这引入了一个新的问题：**测试本身也是 AI 生成的**。如果 AI 生成的测试和 AI 生成的代码都基于同一份 spec，那么测试可能只是在验证"AI 对 spec 的理解是否自洽"，而不是在验证"代码是否真正正确"。

更具体地说：如果 spec 中有一个模糊的表述，AI 在生成代码时会做出某种解释，在生成测试时也会做出同样的解释。测试会通过，但代码的行为可能不是用户真正想要的。

这个问题在 arxiv 论文《Spec-Driven Development: From Code to Contract in the Age of AI Coding》中被明确提出，但目前没有系统性的解决方案。Thoughtworks 的建议是"高度确定性的 CI/CD 实践"，但这更像是一个方向，而不是一个具体的方法。

### 问题四：工具碎片化与生态锁定——SDD 的"标准战争"

2026 年的 SDD 工具生态，呈现出一种令人担忧的碎片化状态。

OpenSpec、GitHub Spec Kit、Amazon Kiro、Tessl——每个工具都有自己的 spec 格式、自己的工作流、自己的命令体系。它们之间不兼容。一个团队如果用 OpenSpec 积累了大量的 spec 文件，想切换到 Kiro，几乎需要从头开始。

这种碎片化带来了几个问题：

**生态锁定风险**：Kiro 绑定了 AWS 和 Claude，Spec Kit 绑定了 GitHub 生态。选择一个工具，就意味着接受它背后的生态绑定。对于企业用户来说，这是一个重大的战略风险。

**最佳实践无法共享**：不同工具的 spec 格式不同，意味着一个团队积累的 spec 写作经验和模板，无法直接迁移到另一个工具。整个行业的学习成本被人为放大。

**"好 spec"的标准缺失**：目前没有任何行业标准来定义什么是"好的 spec"。不同工具对 spec 的要求不同，不同团队的 spec 质量参差不齐。这使得 SDD 的效果高度依赖于团队的具体实践，难以规模化复制。

arxiv 论文中明确指出："there is no universal 'good spec' evaluation yet"（目前还没有通用的"好 spec"评估标准）。这不只是一个学术问题，它直接影响到 SDD 能否从个人实践演变为团队级、组织级的工程能力。

### 问题五：认知负担的转移而非消除——SDD 并没有让开发变得更简单

SDD 的支持者常常把它描述为"让 AI 编程更可靠"的解决方案。但一个经常被忽视的问题是：**SDD 并没有降低总体认知负担，它只是把认知负担从"写代码"转移到了"写规范"**。

在 Vibe Coding 中，开发者的认知负担主要在于"如何描述需求让 AI 理解"和"如何验证 AI 生成的代码"。在 SDD 中，认知负担变成了"如何写出足够精确、足够完整、足够无歧义的规范"。

对于有经验的开发者来说，这种转移是有价值的——写规范比写代码更接近"设计"，更能发挥人类的判断力。但对于经验不足的开发者，写一份好的规范可能比直接写代码更难。

阿里技术团队在内部实践报告中坦承了这个问题：SDD 的"标准版"（Team Mode）需要完整的 Requirements → Interface → Implementation 链路，这对团队的工程能力有相当高的要求。他们提供了一个"Lite 版"（Vibe Mode）作为降级方案，但 Lite 版的效果和 Vibe Coding 的差距并不大。

更深层的问题是：**SDD 假设人类能够在编码之前把所有需求想清楚**。但软件开发的现实是，很多需求只有在实现过程中才会变得清晰。"先写规范再写代码"的线性流程，与软件开发的探索性本质存在根本性的张力。

这不是说 SDD 是错的，而是说它更适合需求相对明确的场景（功能迭代、接口实现），而不适合需求模糊的探索性场景（新产品原型、技术验证）。但目前的 SDD 工具和方法论，往往没有清晰地区分这两种场景。

## 四、竞品对比：SDD 生态中的三条路线

### OpenSpec：开源通用，但深度有限

OpenSpec 是目前最通用的 SDD 框架，MIT 协议，支持 20+ AI 工具。它的核心优势是"低侵入性"——三条命令就能上手，不需要改变现有的开发工具链。

但 OpenSpec 的问题也很明显：它的"通用性"是以"深度"为代价的。因为要兼容所有工具，它无法深度集成任何一个工具的特性。它的 spec 格式是纯 Markdown，没有机器可读的结构化约束，这意味着 spec 的质量完全依赖于写作者的能力，工具本身无法提供任何质量保证。

用户社区的反馈是：OpenSpec 对于个人开发者和小团队很好用，但在大型团队中，缺乏强制性的质量门禁会导致 spec 质量参差不齐，最终退化成"有 spec 格式的 Vibe Coding"。

### GitHub Spec Kit：标准化强，但学习曲线陡

GitHub Spec Kit 的设计哲学是"严格阶段门"——每个阶段必须完成才能进入下一阶段。这使得它在团队协作和可追溯性方面表现出色，但也带来了较高的学习曲线和使用摩擦。

它的另一个问题是主要针对新项目设计，对于已有代码库（Brownfield Projects）的支持不够好。在实际工程中，大多数开发工作是在已有系统上迭代，而不是从零开始。

### Amazon Kiro：集成度最高，但生态锁定最严重

Kiro 是三者中集成度最高的，把 spec 工作流直接嵌入 IDE，体验最流畅。但它的问题是最严重的生态锁定：绑定 Kiro IDE + Claude 模型 + AWS 生态。对于已经有既定工具链的团队，迁移成本极高。

Kiro 的用户反馈两极分化：喜欢它的人觉得它是目前体验最好的 SDD 工具；不喜欢它的人觉得它是一个精心设计的 AWS 生态锁定陷阱。

### Tessl：最激进的赌注

Tessl 走的是最激进的路线：spec 是唯一维护的产物，代码完全由 spec 生成，不需要人工维护代码库。这是 SDD 理念的逻辑终点。

但 Tessl 目前还处于非常早期的阶段，它面临的挑战是：当代码完全由 spec 生成时，如何处理性能优化、调试、遗留系统集成等需要直接操作代码的场景？这些问题目前没有答案。

### 生态位判断

| 维度 | OpenSpec | GitHub Spec Kit | Amazon Kiro | Tessl |
|------|----------|-----------------|-------------|-------|
| 工具绑定 | 无（支持 20+） | GitHub 生态 | Kiro IDE + Claude | 独立 |
| 适用场景 | 新旧项目均可 | 主要新项目 | 新项目 | 早期探索 |
| 学习曲线 | 极低 | 较高 | 中等 | 高 |
| 质量门禁 | 弱（依赖自律） | 强（强制阶段门） | 中等 | 待定 |
| 开源 | MIT | MIT | 闭源 | 闭源 |

目前没有一个工具能够同时做到"低摩擦"和"高质量保证"。这个矛盾是 SDD 工具生态的核心张力，也是未来竞争的主战场。

## 五、综合判断：SDD 是正确方向，但还没到成熟阶段

SDD 的崛起是真实的，它解决的问题也是真实的。在 AI 编程从"个人玩具"走向"团队工程"的过程中，某种形式的规范化是必然的。SDD 提供了一个合理的框架来思考这个问题。

但把 SDD 描述为"AI 编程的终极解决方案"，是过度乐观的。

**它解决了上下文管理的问题，但没有解决规范质量的问题。** 一份好的 spec 需要深厚的工程经验，而这恰恰是很多 AI 编程用户所缺乏的。SDD 把认知负担从"写代码"转移到了"写规范"，但没有降低总体的认知门槛。

**它提供了一个框架，但没有提供验证机制。** 规范和代码之间的一致性验证，目前完全依赖人工审查和测试。在 AI 生成代码的速度远超人工审查速度的情况下，这个瓶颈会越来越明显。

**它的工具生态还处于"战国时代"。** 没有统一的标准，没有互操作性，没有"好 spec"的评估基准。这使得 SDD 的效果高度依赖于团队的具体实践，难以规模化。

**它与软件开发的探索性本质存在张力。** "先写规范再写代码"的线性流程，更适合需求明确的场景，而不适合需求模糊的探索性场景。但目前的 SDD 方法论往往没有清晰地区分这两种场景。

从历史的角度看，SDD 很像 2000 年代的 BDD（行为驱动开发）——理念正确，工具也在快速成熟，但真正大规模落地需要更长的时间，需要工具链的成熟、最佳实践的沉淀，以及开发者认知的转变。

**最可能的未来走向**：SDD 不会取代 Vibe Coding，而是会与它共存，形成一个"分层"的 AI 编程生态——探索性工作用 Vibe Coding，生产级工作用 SDD。工具会逐渐融合，最终的形态可能是"智能 spec 辅助"——AI 不只是按 spec 生成代码，还能主动检测 Spec Drift、评估 spec 质量、在 spec 和代码之间建立双向同步。

但这个未来还需要时间。2026 年的 SDD，是一个正确方向上的早期实践，而不是一个已经解决了核心问题的成熟范式。
