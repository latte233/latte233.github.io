## 第一章：GitNexus 是什么

GitNexus 将代码仓库解析为 AST（抽象语法树）知识图谱，在本地建立符号、调用链、执行流的结构化索引。你可以用自然语言搜索执行流、做变更影响面分析、跨多仓追踪依赖关系。它通过 MCP（Model Context Protocol）协议以 stdio JSON-RPC 方式暴露工具接口，让 Claude Code、Codex、Cursor 等 AI 编码助手直接查询你的代码结构——不是文本匹配，是语义级别的理解。

核心能力一句话总结：**grep 只搜文本，GitNexus 搜结构**。

## 第二章：安装与环境检查

### 2.1 安装

```bash
# macOS (Homebrew)
brew install gitnexus

# 或通过 npm 全局安装
npm install -g gitnexus
```

### 2.2 确认版本

```bash
gitnexus --version
# 输出示例: 1.6.5
```

### 2.3 环境诊断

```bash
gitnexus doctor
```

输出示例：

```

  Node:      v26.3.0
  GitNexus:  1.6.5
  LadybugDB: unknown
  ONNX:      1.26.0

Capabilities
```

关注三个 `available`——说明图存储、全文检索、向量索引全部就绪。如果任一项显示 unavailable，检查 Node.js 版本（≥18）和系统依赖。

### 2.4 .gitnexusignore

在仓库根目录创建 `.gitnexusignore`，语法与 `.gitignore` 完全一致，支持取反：

```gitignore
# 排除生成代码
generated/
**/proto_gen/

# 但保留测试（默认被过滤）
!__tests__/
```

## 第三章：索引构建（analyze）

这是 GitNexus 的核心操作——把代码变成可查询的知识图谱。

### 3.1 基础构建

```bash
cd /path/to/your-repo
gitnexus analyze
```

在仓库根目录执行即可。GitNexus 会自动发现 git root、解析所有源文件、建立符号表和调用关系图。产物存储在 `.gitnexus/` 目录下，同时注册到全局 `~/.gitnexus/registry.json`。

### 3.2 带语义搜索（嵌入向量）

```bash
gitnexus analyze --embeddings
```

启用后，每个符号节点会生成本地嵌入向量（ONNX 推理，不出网），支持自然语言模糊搜索。适合大仓库中"我记得有个处理退款的方法但忘了叫什么"的场景。

限制安全上限（默认 50000 节点）：

```bash
gitnexus analyze --embeddings 100000  # 放宽到 10 万
gitnexus analyze --embeddings 0       # 无上限
```

### 3.3 强制重建

```bash
gitnexus analyze --force
```

即使索引是最新的也重新构建。用于索引损坏或升级 GitNexus 版本后的完整重建。

### 3.4 纯索引模式

```bash
gitnexus analyze --index-only
```

只建立图谱索引，不向仓库注入 `AGENTS.md`、`CLAUDE.md`、skill 文件。适合不想让 GitNexus 修改工作区的场景。

### 3.5 自定义别名

```bash
gitnexus analyze --name my-service
```

当你有多个同名目录（比如都叫 `app`）时，用别名区分。后续 CLI 查询用 `-r my-service` 指定目标。

### 3.6 跳过 AI 上下文注入的各级开关

| Flag | 效果 |
|------|------|
| `--index-only` | 跳过所有文件注入（AGENTS.md + CLAUDE.md + skills） |
| `--skip-agents-md` | 仅跳过 AGENTS.md 和 CLAUDE.md |
| `--skip-skills` | 仅跳过 `.claude/skills/gitnexus/` 标准技能文件 |
| `--no-stats` | AGENTS.md 中不写入易变的文件/符号数量 |

### 3.7 环境变量调优

```bash
# 跳过超过 1MB 的文件（默认 512KB）
GITNEXUS_MAX_FILE_SIZE=1024 gitnexus analyze

# Worker 超时延长到 60 秒（默认 30 秒，大仓库可能需要）
GITNEXUS_WORKER_SUB_BATCH_TIMEOUT_MS=60000 gitnexus analyze

# 限制嵌入线程数（避免 CPU 跑满）
GITNEXUS_EMBEDDING_THREADS=2 gitnexus analyze --embeddings
```

### 3.8 查看索引状态

```bash
# 当前仓库状态
gitnexus status

# 全局已注册仓库列表
gitnexus list
```

### 3.9 手动注册已有索引

如果仓库中已有 `.gitnexus/` 目录（比如从另一台机器拷贝过来），可以直接注册而不重新分析：

```bash
gitnexus index /path/to/repo-with-existing-index
```

## 第四章：CLI 查询（单仓）

所有查询命令共享 `-r, --repo <name>` 参数指定目标仓库。如果全局只注册了一个仓库，可以省略。

### 4.1 query — 按概念搜索执行流

**用途**：用自然语言描述你想找的功能，返回相关的执行流（方法调用链）。

```bash
gitnexus query "用户下单后扣减库存"
```

带上下文提升精度：

```bash
gitnexus query "库存扣减" \
  -c "正在排查超卖问题" \
  -g "找到库存校验和扣减的完整链路"
```

参数说明：

| 参数 | 作用 |
|------|------|
| `-c, --context` | 你当前在做什么（背景） |
| `-g, --goal` | 你想找到什么（目标） |
| `-l, --limit <n>` | 返回结果数量，默认 5 |
| `--content` | 附带完整源代码 |

### 4.2 context — 符号 360° 视图

**用途**：给定一个符号名，展示它的调用者、被调用者、所属执行流。

```bash
gitnexus context OrderService.createOrder
```

按文件精确定位（同名方法太多时）：

```bash
gitnexus context createOrder -f src/service/OrderService.java
```

按 UID 精确查找（零歧义）：

```bash
gitnexus context -u "method:src/service/OrderService.java:createOrder:45"
```

加 `--content` 看完整源码：

```bash
gitnexus context OrderService.createOrder --content
```

### 4.3 impact — 变更爆炸半径

**用途**：如果我改了这个符号，哪些地方会受影响？

```bash
# 谁依赖这个方法（上游 = 被谁调用）
gitnexus impact OrderService.createOrder

# 这个方法依赖什么（下游）
gitnexus impact OrderService.createOrder --direction downstream

# 加大遍历深度
gitnexus impact OrderService.createOrder --depth 5

# 包含测试文件
gitnexus impact OrderService.createOrder --include-tests
```

### 4.4 detect-changes — 把 git diff 映射到符号

**用途**：当前改动影响了哪些符号？哪些执行流可能被破坏？

```bash
# 分析未暂存的改动
gitnexus detect-changes

# 分析已暂存的改动
gitnexus detect-changes --scope staged

# 分析所有改动（staged + unstaged）
gitnexus detect-changes --scope all

# 与某个分支对比
gitnexus detect-changes --scope compare --base-ref main
```

典型场景：提交前检查影响面，或 code review 时快速理解一个 PR 的波及范围。

### 4.5 cypher — 原始图查询

**用途**：当预置命令不够用时，直接写 Cypher 查询知识图谱。

```bash
# 找所有调用了 Redis 的方法
gitnexus cypher "MATCH (n)-[:CALLS]->(m) WHERE m.name CONTAINS 'Redis' RETURN n.name, m.name LIMIT 20"

# 找入度最高的 10 个符号（最多人调用的）
gitnexus cypher "MATCH (n)<-[:CALLS]-(caller) RETURN n.name, count(caller) AS fanIn ORDER BY fanIn DESC LIMIT 10"
```

### 4.6 augment — 搜索增强（Hook 用）

**用途**：给一个搜索 pattern 附加知识图谱上下文，通常由编辑器 hook 自动调用，很少手动使用。

```bash
gitnexus augment "handlePayment"
```

## 第五章：跨仓协作（Group）

当你的系统由多个微服务仓库组成时，单仓分析无法追踪跨仓调用链。Group 功能解决这个问题。

### 5.1 建组三步走

```bash
# 1. 创建组
gitnexus group create my-platform

# 2. 添加成员（已在 gitnexus list 中注册的仓库）
gitnexus group add my-platform "service/order" order-service
gitnexus group add my-platform "service/inventory" inventory-service
gitnexus group add my-platform "gateway/api" api-gateway

# 3. 同步契约（提取接口定义、建立跨仓链接）
gitnexus group sync my-platform
```

`group add` 三个参数的含义：
- `my-platform`：组名
- `"service/order"`：层级路径（自定义分类，用于子组过滤）
- `order-service`：registry 中的仓库名（`gitnexus list` 中显示的名字）

### 5.2 跨仓查询

```bash
gitnexus group query my-platform "用户下单完整链路"

# 限制搜索范围到某个子组
gitnexus group query my-platform "库存扣减" --subgroup "service/"
```

### 5.3 跨仓影响面

```bash
gitnexus group impact my-platform \
  --target "InventoryService.deduct" \
  --repo "service/inventory" \
  --direction upstream \
  --cross-depth 2
```

这会告诉你：如果改了 inventory 仓库里的 `deduct` 方法，跨仓最多追 2 跳，上游有哪些仓库的哪些方法会受影响。

### 5.4 查看契约注册表

```bash
# 所有契约
gitnexus group contracts my-platform

# 按类型过滤
gitnexus group contracts my-platform --type rpc

# 只看未匹配的（可能是死接口）
gitnexus group contracts my-platform --unmatched
```

### 5.5 组管理

```bash
# 列出所有组
gitnexus group list

# 查看某个组的详情
gitnexus group list my-platform

# 检查新鲜度（哪些仓库索引过期了）
gitnexus group status my-platform
```

### 5.6 实战案例：营销域多仓联动

```bash
# 创建营销组
gitnexus group create promotion

# 添加所有营销相关仓库
gitnexus group add promotion "core/promotion" mall-promotion
gitnexus group add promotion "api/mapi" mall-promotion-mapi
gitnexus group add promotion "admin/management" mall-promotion-management
gitnexus group add promotion "biz/salespromotion" mall-promotion-salespromotion
gitnexus group add promotion "recall/engine" groceryretail-promotion-recall

# 同步契约
gitnexus group sync promotion

# 查询：创建促销活动的完整链路
gitnexus group query promotion "创建促销活动"

# 影响面：改了核心促销引擎，谁会挂？
gitnexus group impact promotion \
  --target "PromotionEngine.calculate" \
  --repo "core/promotion" \
  --direction upstream
```

## 第六章：MCP Server 模式

### 6.1 什么是 MCP

MCP（Model Context Protocol）是 Anthropic 提出的协议标准，让 AI 模型通过 stdio JSON-RPC 调用外部工具。GitNexus 实现了 MCP server，AI 编码助手启动 `gitnexus mcp` 进程后，就能直接调用查询/分析能力。

### 6.2 启动 MCP Server

```bash
gitnexus mcp
```

这是一个 stdio 进程——不监听端口，通过 stdin/stdout 通信。由宿主程序（Claude Code / Codex / Cursor）负责启动和管理它的生命周期。

### 6.3 MCP 暴露的工具

MCP 模式下暴露的工具与 CLI 命令一一对应：

| MCP Tool | 等价 CLI | 用途 |
|----------|----------|------|
| `gitnexus_query` | `gitnexus query` | 执行流搜索 |
| `gitnexus_context` | `gitnexus context` | 符号 360° 视图 |
| `gitnexus_impact` | `gitnexus impact` | 爆炸半径 |
| `gitnexus_detect_changes` | `gitnexus detect-changes` | diff → 符号映射 |
| `gitnexus_cypher` | `gitnexus cypher` | 原始 Cypher |

### 6.4 MCP vs CLI 的选择

- **MCP**：AI 工具自动调用，无需人工介入，是日常开发的默认推荐方式
- **CLI**：人工调试、验证结果、写脚本自动化时使用

两者共享同一份索引，结果完全一致。

## 第七章：对接 Claude Code

### 7.1 自动注册（推荐）

```bash
gitnexus setup
```

这条命令会自动检测系统中安装的 AI 工具，将 MCP 配置写入对应的配置文件。对 Claude Code 来说，写入 `~/.claude.json`。

### 7.2 手动注册

如果自动注册不生效，手动编辑 `~/.claude.json`：

```json
{
  "mcpServers": {
    "gitnexus": {
      "command": "/opt/homebrew/bin/gitnexus",
      "args": ["mcp"]
    }
  }
}
```

如果通过 npm 安装，command 路径改为 `npx` 或 `gitnexus` 的实际路径（`which gitnexus` 查看）。

也可以做项目级配置，在仓库根目录创建 `.mcp.json`：

```json
{
  "mcpServers": {
    "gitnexus": {
      "command": "gitnexus",
      "args": ["mcp"]
    }
  }
}
```

### 7.3 权限放行

Claude Code 首次调用 MCP 工具时会弹出权限确认。如需自动放行，编辑 `~/.claude/settings.json`：

```json
{
  "permissions": {
    "allow": ["mcp__gitnexus__*"]
  }
}
```

### 7.4 验证

在 Claude Code 中输入：

```
帮我查一下 OrderService.createOrder 的调用方和被调用方
```

如果 Claude Code 调用了 `gitnexus_context` 工具并返回结果，说明集成成功。

### 7.5 AGENTS.md 和 CLAUDE.md

`gitnexus analyze` 默认会在仓库中注入 `AGENTS.md` 和 `.claude/CLAUDE.md`，包含仓库结构摘要和 GitNexus 工具使用指引。Claude Code 启动时自动读取这些文件，让模型知道可以用 GitNexus 查代码结构。

如果不需要：

```bash
gitnexus analyze --skip-agents-md
```

## 第八章：对接 Codex

### 8.1 配置 MCP Server

编辑 `~/.codex/config.toml`，在 `[mcp_servers]` 区块添加：

```toml
[mcp_servers.gitnexus]
command = "/opt/homebrew/bin/gitnexus"
args = ["mcp"]
```

### 8.2 启动超时

如果仓库索引较大，首次加载可能较慢，可设置超时：

```toml
[mcp_servers.gitnexus]
command = "/opt/homebrew/bin/gitnexus"
args = ["mcp"]
startup_timeout_sec = 60
```

### 8.3 与沙箱策略的关系

Codex 的沙箱模式（suggest / auto-edit / full-auto）不影响 MCP 工具调用——MCP 工具只是查询，不修改文件，在任何策略下都可正常使用。

### 8.4 验证

在 Codex 会话中：

```
用 gitnexus 帮我查一下 handlePayment 的影响面
```

Codex 会自动调用 `gitnexus_impact` 工具。

## 第九章：对接 Cursor / OpenCode / CatDesk

### 9.1 Cursor

在项目根目录创建 `.cursor/mcp.json`：

```json
{
  "mcpServers": {
    "gitnexus": {
      "command": "gitnexus",
      "args": ["mcp"]
    }
  }
}
```

或全局配置：`~/.cursor/mcp.json`，格式相同。

### 9.2 OpenCode

OpenCode 原生支持 MCP，运行 `gitnexus setup` 时会自动注册。手动配置参考其 `~/.opencode/config.json` 中的 `mcpServers` 字段。

### 9.3 CatDesk

CatDesk 通过项目级 `mcps/` 目录管理 MCP 服务。在项目 `.catpaw/` 下创建对应的 tool descriptor JSON 文件即可。`gitnexus setup` 会自动处理。

### 9.4 通用 MCP 配置模板

所有支持 MCP 的工具，核心配置都是同一个模式：

```json
{
  "command": "<gitnexus的绝对路径>",
  "args": ["mcp"],
  "transport": "stdio"
}
```

找到你工具的 MCP 配置入口，把这段填进去就行。

## 第十章：Wiki 生成与发布

### 10.1 生成仓库 Wiki

```bash
gitnexus wiki
```

从知识图谱自动生成结构化的仓库文档。需要配置 LLM 提供者。

### 10.2 配置 LLM

```bash
# 使用 OpenAI 兼容接口
gitnexus wiki --provider openai --base-url https://api.example.com/v1 --api-key sk-xxx

# 使用 Azure OpenAI
gitnexus wiki --provider openai \
  --base-url "https://myresource.openai.azure.com/openai/v1" \
  --api-key "your-azure-key" \
  --api-version "2024-10-21"

# 使用推理模型（o1/o3/o4-mini）
gitnexus wiki --reasoning-model --model o4-mini
```

配置会保存到 `~/.gitnexus/config.json`，后续执行无需重复传参。

### 10.3 调优参数

```bash
gitnexus wiki \
  --concurrency 5 \
  --timeout 120 \
  --retries 5
```

### 10.4 先审后生成

```bash
gitnexus wiki --review
```

执行到模块分组阶段暂停，让你审查模块划分结构，确认后再继续生成。

### 10.5 发布到 GitHub Gist

```bash
gitnexus wiki --gist
```

生成完成后自动创建公开 Gist，返回链接。

## 第十一章：维护与清理

### 11.1 删除当前仓库索引

```bash
cd /path/to/repo
gitnexus clean        # 有确认提示
gitnexus clean -f     # 跳过确认
```

### 11.2 删除指定仓库索引

不需要进入仓库目录，通过名字或路径远程删除：

```bash
gitnexus remove my-service
gitnexus remove /path/to/old-repo
```

### 11.3 清除所有索引

```bash
gitnexus clean --all
```

### 11.4 Web UI 调试

```bash
gitnexus serve              # 默认 http://127.0.0.1:4747
gitnexus serve -p 8080      # 自定义端口
gitnexus serve --host 0.0.0.0  # 局域网可访问
```

启动后在浏览器打开，可视化浏览知识图谱。

## 附录 C：常见问题 FAQ

**Q: 索引构建太慢怎么办？**

大仓库（>10 万文件）建议先用 `.gitnexusignore` 排除不需要的目录（vendor、node_modules、生成代码），然后适当调大 `GITNEXUS_WORKER_SUB_BATCH_TIMEOUT_MS`。嵌入生成是最耗时的步骤，如果不需要语义搜索，去掉 `--embeddings` 会快很多。

**Q: MCP 连不上？**

检查三件事：1）`which gitnexus` 确认路径正确；2）配置文件中的 command 使用绝对路径；3）Claude Code / Codex 重启后才会重新加载 MCP 配置。用 `gitnexus mcp` 手动启动测试是否报错。

**Q: 跨仓 sync 报 stale？**

说明某个成员仓库的索引版本过旧。运行 `gitnexus group status <name>` 找到过期的仓库，进入该仓库执行 `gitnexus analyze` 更新索引，然后重新 `gitnexus group sync`。

**Q: 多人协作时索引要提交到 git 吗？**

不推荐。`.gitnexus/` 目录应加入 `.gitignore`。每人本地构建自己的索引，因为索引依赖本机绝对路径。共享的是 `.gitnexusignore` 配置。

**Q: 支持哪些语言？**

GitNexus 基于 tree-sitter 解析，支持主流语言：Java、TypeScript/JavaScript、Python、Go、Rust、C/C++、Ruby、PHP、Kotlin、Swift 等。具体覆盖度取决于 tree-sitter 语法包版本。

**Q: 索引占多大空间？**

约为源代码体积的 0.5-2 倍。带嵌入的约 2-5 倍。一个 10 万行 Java 项目，索引约 50-100MB。
