Claude Code 已经在生产环境中运行于数百万行代码的单体仓库、数十年历史的遗留系统、跨越数十个仓库的分布式架构，以及拥有数千名开发者的组织中。这些环境带来了小型、简单代码库所没有的挑战——比如每个子目录都有不同的构建命令，或者遗留代码散布在没有共享根目录的文件夹中。

本文涵盖了我们观察到的、在大规模部署 Claude Code 方面取得成功的模式。我们使用"大型代码库"来指代各种部署场景：包含数百万行的单体仓库、数十年构建的遗留系统、跨多个独立仓库的数十个微服务，或以上任何组合。这也包括使用某些语言运行的代码库，这些语言团队并不总是将其与 AI 编程工具关联起来，比如 C、C++、C#、Java、PHP。（Claude Code 在这些场景中的表现比大多数团队预期的要好，尤其是最近的模型版本。）虽然每个大型代码库的部署都受其特定的版本控制、团队结构和积累的约定影响，但这里的模式可以跨场景通用，是考虑采用 Claude Code 的团队的不错起点。

## Claude Code 如何导航大型代码库

Claude Code 导航代码库的方式就像软件工程师一样：它遍历文件系统、读取文件、使用 grep 精确定位所需内容，并在代码库中追踪引用。它在开发者的本地机器上运行，不需要构建、维护或上传代码库索引到服务器。

基于 RAG 的 AI 编程工具通过嵌入整个代码库并在查询时检索相关片段来工作。在大规模场景下，这些系统可能会失败，因为嵌入管道无法跟上活跃的工程团队。当开发者查询索引时，它反映的是代码库在数周、数天甚至数小时前的状态。检索返回的是团队两周前重命名的函数，或者引用的是上一个冲刺中被删除的模块，而没有任何迹象表明两者都已过时。

智能体搜索避免了这些失败模式。没有嵌入管道或集中式索引需要维护，因为成千上万的工程师在提交新代码。每个开发者的实例都基于实时代码库运行。

但这种方法有权衡：它在 Claude 有足够的起始上下文来知道该往哪里看时效果最好。这意味着 Claude 的导航质量取决于代码库的设置方式，通过 CLAUDE.md 文件和 skills 来分层上下文。如果你要求它在数十亿行代码库中找到所有模糊模式的实例，你会在工作开始前就触及上下文窗口限制。那些在代码库设置上投入的团队会看到更好的结果。

## 工具链（harness）与模型同样重要

关于 Claude Code 最常见的误解之一是，它的 capabilities 完全由使用的模型决定。团队关注模型的基准测试和它在测试任务上的表现。在实践中，围绕模型构建的工具链——即 harness——比单独的模型更能决定 Claude Code 的表现。

工具链由五个扩展点构建而成——CLAUDE.md 文件、hooks、skills、plugins 和 MCP servers——每个都有不同的功能。团队构建它们的顺序很重要，因为每一层都建立在前一层的基础上。另外两个功能——LSP 集成和子智能体（subagents）——完成了整个设置。下面，我们解释每个组件和功能的作用：

**CLAUDE.md 文件排在第一位**。这些是 Claude 在每个会话开始时自动读取的上下文文件：根文件提供全局视图，子目录文件提供本地约定。它们赋予 Claude 所需的代码库知识，使其能够出色地完成任何工作。因为它们在每个会话中都会加载，无论任务是什么，所以让它们专注于广泛适用的内容，可以防止它们成为性能负担。

**Hooks 使设置能够自我改进**。大多数团队将 hooks 视为防止 Claude 做错事的脚本，但它们更有价值的使用场景是持续改进。stop hook 可以在会话期间反思发生的事情，并在上下文还很新鲜时提出 CLAUDE.md 更新建议。start hook 可以动态加载团队特定的上下文，这样每个开发者都不需要手动配置，就能为其模块获得正确的设置。对于 lint 和格式化等自动化检查，hooks 以确定性方式执行规则，产生比依赖 Claude 记住指令更一致的结果。

**Skills 让正确的专业知识按需可用，而不会让每个会话都臃肿**。在包含数十种任务类型的大型代码库中，并非所有专业知识都需要在每个会话中都存在。Skills 通过渐进式披露（progressive disclosure）解决这个问题，将专业化的工作流程和领域知识卸载出去，否则它们会竞争上下文空间，只在任务需要时才加载。例如，当 Claude 评估代码漏洞时，安全审查 skill 会被加载；而当进行代码更改需要更新文档时，文档处理 skill 会被加载。

Skills 也可以绑定到特定路径，这样它们只会在代码库的相应部分激活。负责支付服务的团队可以将他们的部署 skill 绑定到该目录，这样当有人在单体仓库的其他地方工作时，它永远不会自动加载。

**Plugins 分发有效的设置**。大型代码库面临的挑战之一是好的设置可能只是小圈子里的东西。Plugin 将 skills、hooks 和 MCP 配置捆绑到一个可安装的包中，这样当新工程师在第一天安装该插件时，他们将立即拥有与已经使用 Claude 的人相同的上下文和 capabilities。Plugin 更新可以通过托管市场在组织内部分发。

例如，与我们合作的一个大型零售组织构建了一个将 Claude 连接到其内部分析平台的 skill，这样业务分析师就可以在不离开工作流程的情况下拉取性能数据。在向业务部门广泛推广之前，他们将其作为插件分发。

**语言服务器协议（LSP）集成赋予 Claude 与开发者在 IDE 中相同的导航能力**。大多数大型代码库的 IDE 已经运行着 LSP，为"跳转到定义"和"查找所有引用"提供支持。向 Claude 暴露这些功能赋予它符号级别的精确度：它可以追踪函数调用到其定义，追踪跨文件的引用，并区分不同语言中的同名函数。没有它，Claude 会基于文本进行模式匹配，可能定位到错误的符号。我们合作过的一家企业管理软件公司在 Claude Code 推广之前就在全公司部署了 LSP 集成，专门为了在大规模场景下使 C 和 C++ 的导航可靠。对于多语言代码库，这是最高价值的投资之一。

**MCP servers 扩展一切**。MCP servers 是 Claude 连接到它无法直接访问的内部工具、数据源和 API 的方式。最先进的团队构建了暴露结构化搜索作为 Claude 可以直接调用的工具的 MCP servers。其他人将 Claude 连接到内部文档、工单系统或分析平台。

**子智能体（Subagents）将探索与编辑分离**。子智能体是一个隔离的 Claude 实例，有自己的上下文窗口，接收任务、执行工作，然后只将最终结果返回给父智能体。一旦工具链就位，一些团队会启动一个只读子智能体来映射子系统并将发现写入文件，然后让主智能体在完整图景下进行编辑。

下表总结了每个组件的作用、加载时机以及我们看到的最常见错误：

| 组件 | 它是什么 | 加载时机 | 最适合 | 常见困惑 |
|---|---|---|---|---|
| CLAUDE.md | Claude 自动读取的上下文文件 | 每个会话 | 项目特定约定、代码库知识 | 将应该放在 skill 中的可重用专业知识用于此 |
| Hooks | 在关键时刻运行的脚本 | 由事件触发 | 自动化一致行为、捕获会话学习 | 用 prompts 做应该自动运行的事情 |
| Skills | 针对特定任务类型的打包指令 | 按需，当相关时 | 跨会话和项目的可重用专业知识 | 将所有内容加载到 CLAUDE.md 而不是用 skills |
| Plugins | 捆绑的 skills、hooks、MCP 配置 | 配置后始终可用 | 在组织内部分发有效的设置 | 让好的设置保持小圈子状态 |
| 语言服务器协议 (LSP)* | 通过语言特定服务器提供的实时代码智能 | 配置后始终可用 | 符号级导航和类型语言的自动错误检测 | 假设它是自动的 |
| MCP servers | 连接到外部工具和数据 | 配置后始终可用 | 赋予 Claude 访问无法直接访问的内部工具 | 在基础功能还没工作好之前就构建 MCP 连接 |
| Subagents* | 用于特定任务的独立 Claude 实例 | 调用时 | 将探索与编辑分离、并行工作 | 在同一会话中进行探索和编辑 |

*LSP 通过插件层访问。Subagents 是一种委托能力，而不是配置的扩展点。

## 来自成功部署的三种配置模式

如何为大型代码库配置 Claude Code 在很大程度上取决于该代码库的结构。不过，我们观察到的部署中有三种模式始终出现。

### 使代码库在大规模下可导航

Claude 在大型代码库中提供帮助的能力受限于它找到正确上下文的能力。每次会话加载太多上下文会降低性能，而上下文太少则让 Claude 在黑暗中摸索。最有效的部署在让代码库对 Claude 可读方面进行前期投入。几个模式始终出现：

- **保持 CLAUDE.md 文件精简和分层**。Claude 在代码库中移动时会累加加载它们：根文件提供全局视图，子目录文件提供本地约定。根文件应该只包含指针和关键的注意事项；其他一切都会沦为噪音。

- **在子目录中初始化，而不是在仓库根目录**。当 Claude 被限定在实际与任务相关的代码库部分时，它效果最好。在单体仓库中，这可能感觉违反直觉，因为工具通常假设根目录访问，但 Claude 会自动向上遍历目录树并加载沿途找到的每个 CLAUDE.md 文件，所以根级别的上下文永远不会丢失。

- **按子目录限定测试和 lint 命令**。当 Claude 只更改了一个服务时运行完整套件会导致超时，并浪费上下文在无关输出上。子目录级别的 CLAUDE.md 文件应该指定适用于该部分代码库的命令。这对于每个目录都有自己测试和构建命令的服务导向型代码库效果很好。在具有深层跨目录依赖的编译语言单体仓库中，按子目录限定更难实现，可能需要特定项目的构建配置。

- **使用 `.claude`ignore` 文件排除生成的文件、构建产物和第三方代码**。在 `.claude/settings.json` 中提交 `permissions.deny` 规则意味着排除项是版本控制的，所以团队中的每个开发者都无需自己配置就能获得相同的降噪效果。在某些代码库中，生成的文件本身就是开发工作的主题。从事代码生成器工作的开发者可以在本地设置中覆盖项目级别的排除项，而不会影响团队其他成员。

- **当目录结构不能胜任时，构建代码库地图**。对于代码不集中在传统目录结构中的组织，在仓库根目录放置一个轻量级 markdown 文件，列出每个顶级文件夹及其包含内容的一行描述，让 Claude 可以在打开文件之前扫描的目录表。对于有数百个顶级文件夹的代码库，这最好作为分层方法：根文件只描述最高级别的结构，子目录的 CLAUDE.md 文件提供下一级细节，在 Claude 在树中移动时按需加载。对于更简单的情况，@-mentioning 特定文件或目录也可以完成同样的工作。

- **运行 LSP 服务器让 Claude 按符号而非字符串搜索**。在大型代码库中 grep 一个常见函数名会返回数千个匹配，Claude 会燃烧上下文打开文件来弄清楚哪个是重要的。LSP 只返回指向同一符号的引用，所以过滤发生在 Claude 读取任何内容之前。设置这个需要为你的语言安装代码智能插件和相应的语言服务器二进制文件；Claude Code 文档涵盖了可用的插件和故障排除。

**一个注意事项**：在某些极端情况下，即使分层 CLAUDE.md 方法也会失效，例如具有数十万文件夹和数百万文件的代码库，或使用非 git 版本控制的遗留系统。我们将在本系列的后续文章中解决它们的挑战。

### 随着模型智能的演进主动维护 CLAUDE.md 文件

随着模型的发展，为当前模型编写的指令可能会对未来的模型不利。指导 Claude 度过它曾经挣扎的模式的 CLAUDE.md 文件，在下一个模型发布时可能会变得不必要或主动限制。例如，一个告诉 Claude 将每个重构分解为单文件更改的 CLAUDE.md 规则可能帮助过早期模型保持正轨，但会阻止更新模型执行它处理得很好的协调跨文件编辑。

为补偿特定模型限制而构建的 skills 和 hooks——无论是在模型的推理还是 Claude Code 本身的工具中——一旦这些限制不再存在，就成为开销。例如，一个拦截文件写入以在 Perforce 代码库中强制执行 p4 edit 的 hook，一旦 Claude Code 添加了原生 Perforce 模式，就变得冗余了。

团队应该每三到六个月进行一次有意义的配置审查，但在重大模型发布后性能感觉停滞不前时也值得做一次。

### 为 Claude Code 管理和采用指定负责人

仅靠技术配置并不能推动采用。做得好的组织也在组织层面投入了。

推广最快的 rollout 在广泛访问之前就有专门的基础设施投入。一小团队，有时甚至只有一个人，负责将工具连接好，这样当开发者第一次接触 Claude 时，它已经适合开发者的工作流程了。在一家公司，几位工程师构建了一套在第一天就可用的插件和 MCP。在另一家公司，一个专注于管理 AI 编程工具的整个团队在推广开始之前就已经准备好了基础设施。在这两种情况下，开发者的第一次体验是富有成效的，而不是令人沮丧的，然后采用从这里传播开来。

今天从事这项工作的团队往往隶属于开发者体验或开发者生产力部门，这通常是负责新工程师入职和构建开发者工具的职能。在几个组织中，一个新兴角色是 agent manager：一个专注于管理 Claude Code 生态系统的混合 PM/工程师职能。对于没有专门团队的组织，最小可行版本是一个 DRI：一个人拥有 Claude Code 配置的所有权、设置决策的权威、权限策略、插件市场的权限，以及 CLAUDE.md 约定的权力，并负责保持它们的更新。

自下而上的采用会产生热情，但如果没有人集中有效的做法，可能会碎片化。你需要一个人或一个团队来汇编和推广正确的 Claude Code 约定（如标准化的 CLAUDE.md 层级或精选的 skills 和 plugins 集）。没有这项工作，知识将保持小圈子状态，采用将停滞。

在大型组织中，尤其是受监管行业的组织，治理问题很早就出现，比如：谁控制哪些 skills 和 plugins 可用，如何防止数千名工程师独立重建相同的东西，如何确保 AI 生成的代码经过与人类生成的代码相同的审查流程？为了尽早解决这些问题，我们建议从一套明确定义的批准 skills、必需的代码审查流程和有限的初始访问开始，随着信心的建立而扩大。

我们观察到，在组织中最早建立跨职能工作组——汇集工程、信息安全和治理代表共同定义需求并构建推广路线图——部署最为顺利。

## 将这些模式应用到你的组织

Claude Code 围绕传统软件工程环境设计，其中工程师是主要的代码库贡献者，仓库使用 Git，代码遵循标准目录结构。大多数大型代码库符合这个模式，但游戏引擎中有大量二进制资产、环境使用非传统版本控制，或非工程师贡献代码等非传统设置需要额外的配置工作。我们的指导假设传统设置，我们描述的模式已在许多客户中生效。任何剩余的复杂性都需要针对你的代码库、工具和组织的特定判断。这就是 Anthropic 的应用 AI 团队直接与工程团队合作将这些模式转化为组织特定需求的地方。
