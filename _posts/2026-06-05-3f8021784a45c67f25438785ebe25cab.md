> 一句话先说清楚：Multica 是一个开源的「托管式 Agent 平台」，它把 Claude Code、Codex、Cursor 这些编程智能体，从你终端里一次性使唤的工具，变成项目看板上有名字、有档案、会主动汇报进度、能被「@」的正式团队成员。它对标的不是某个编程工具，而是 Linear、Jira 这一层——只不过看板上坐着的，一半是人，一半是 AI。

这篇文章会沿着两条线把 Multica 讲透：一条是时间线，还原它从一个失败的 AI 搜索引擎、一段在 TikTok 的工程师经历，到三个月冲上 GitHub 一万多颗星的完整故事；另一条是横切面，把它和同赛道的 Vibe Kanban、Conductor、Devin 等放在一起比，看清它到底站在赛道的哪个位置。两条线交汇处，是对一个更大问题的回答：当 AI 写代码已经足够好，团队和一群 Agent 协作的「操作系统」该长什么样。

## 一、它是怎么来的：一个 95 后、三次 pivot，和一次对 Multics 的致敬

### 从步步高学习机开始的技术极客

Multica 的故事，绕不开它的创始人张佳圆（Jiayuan Zhang）。

按他自己在采访里的说法，他大概在初中、也就是 15 年前开始自学编程，最初的动机相当朴素——想给一台学习机写游戏，接触的第一门语言是 BBK Basic。这种「为了玩做点东西」的起点，很大程度上塑造了他后来做产品的路径：先解决自己真实遇到的问题，再想要不要做成一个产品。

工作上，他曾在字节跳动旗下的 TikTok 做工程师。2023 年前后，他离开大厂开始创业，方向一直锁定在「面向开发者的工具」这条线上。值得注意的是一个细节：他不只是个埋头写代码的人，在推特上他还是个有一定影响力的技术博主，写过一系列关于 RAG（检索增强生成）的分享，传播很广。这一点后来会反复在 Multica 身上体现——一个懂技术、又懂怎么讲故事、怎么经营开发者社区的创始人，做出来的产品天然带传播基因。

### Devv 和 DevCode：两个「黄掉」的前作

Multica 是张佳圆的第三个产品。在它之前，有两个绕不过去的前作，而且都没能跑通。

第一个是 Devv AI，一个面向开发者的 AI 搜索引擎。这个产品其实一度相当亮眼：按公开报道，它上线四周就做到了周活约 2.5 万人、日搜索约 4 万次、周增长 200%，到 2024 年累计用户达到 50 万级别。对一个早期创业项目来说，这是一份漂亮的成绩单，也让他很快拿到了头部美元基金的投资。

但漂亮的数据没能挡住一个结构性问题：AI 搜索这个品类本身在萎缩。张佳圆后来的复盘很直接——当模型本身越来越强，开发者想查一个函数怎么用，直接问 GPT、问 Claude 就行了，专门做一个「面向开发者的 AI 搜索引擎」这件事的需求空间被通用大模型挤掉了。于是 Devv 逐渐淡出。

第二个是 DevCode（也写作 Devv Code），一个 AI 编程产品。它是顺着 Devv 的用户群往「写代码」这个更核心的环节延伸的尝试。但这个阶段恰好是 Cursor、Claude Code、Codex 等编程 Agent 集体爆发的窗口期——一个小团队想在编程工具本体这条最拥挤的赛道上正面硬刚巨头，几乎不可能。DevCode 同样因为市场环境变化而调整方向。

这两次「失败」非常关键。它们不是白交的学费：连续三年深耕「AI + 开发者工具」这条线，让团队对这个领域的演进积累了一种很难被复制的「手感」——他们清楚地知道模型能力走到了哪一步，知道开发者的真实痛点正在从哪里转移到哪里。这种手感，恰恰是 Multica 能踩准时间点的底层原因。

### 真正的转折：一次被打断 20 分钟的协作

Multica 的诞生时间线其实短得惊人——从想法到爆发，前后不到四个月。张佳圆把它拆成了几个清晰的节点：

2026 年 1 月，「想法第一次出现在 journal 里」。但请注意，那个最初的想法并不是 Multica，而是「做一个更好的本地 Coding Agent」——也就是说，团队一开始还是想继续做 DevCode 那条路。

2026 年 2 月，团队开始做第一版原型，延续的依然是「本地 Agent」的思路。

真正的转折发生在 2 月下旬到 3 月。团队内部先用起了自己的原型，结果撞上了两个谁用谁知道的痛点——「Context 孤岛」和「多 Agent 管理混乱」。

张佳圆讲过一个特别具体的场景，可以说是整个 Multica 的「创世瞬间」：有一周，他同时在推进三个相对独立的功能。按老办法，他会打开本地的 Claude Code，开 3 到 5 个 session 轮流和 Agent 对话，写完一个 review 一个。但麻烦在于，当某个功能需要另一位同事做产品判断时，整条协作链就断了——他得把 Agent 的对话记录截图，发到飞书，等同事看完回复，再把反馈粘回本地终端，Agent 才能继续。

他的原话是：「一个很小的协作动作，能打断我 20 分钟。」

于是一个朴素的疑问冒了出来：为什么 Agent 的协作空间是孤立的本地终端，而人的协作空间是飞书或 Linear？能不能把它们合到同一个空间里？

顺着这个疑问，团队做了第一版原型——一个像 Linear 那样的看板，但参与者同时包括人和 Agent。用了一周后，他们意识到这件事的意义远不止「合并两个协作空间」这么简单：它彻底改变了任务分派的形态——从「人一句一句驱动 Agent」，变成了「批量创建 Issue → Agent 自主执行 → 完成后人来 review 决策」。

这就是从「单人本地 Agent」到「多人多 Agent 协作平台」的方向转身。一个原本要做 DevCode 2.0 的团队，因为自己用着难受，意外撞出了一个全新品类。

### 上线即开源，三个月一万多星

2026 年 4 月 1 日，新产品在创始人个人账号上小范围上线。一个有意思的传播背景是：当时正值整个 AI 编程圈关注度极高的窗口，Multica 几乎是「踩着点」开源亮相，迅速进入了大量开发者的视野。

接下来的数据曲线相当陡峭。多个信源交叉印证：项目于 2026 年 1 月在 GitHub 建立代码仓库，4 月正式发布后开始爆发；一周之内涨了约 1.2 万颗 Star，连续登上 GitHub Trending；到 4 月中下旬，仓库 Star 数超过 2.2 万、Fork 约 2800；用了大约三个月，挤进 GitHub 全球仓库排名前几千名；平台上「平均每 10 秒就有一个 Agent 任务被触发执行」。

而支撑这一切的团队，只有 4 个真人，外加十几个不同的 AI Agent。这个「4 个真人 + 一群 Agent」的配置本身就是 Multica 最好的广告——他们正是用 Multica 在开发 Multica，把自己活成了产品的第一个、也是最有说服力的案例。

融资方面，张佳圆透露团队已完成两轮融资，老股东是头部美元基金，并准备开启新一轮。产品当时尚未商业化，计划在 2026 年 5 月开始尝试商业化。

### 为什么叫 Multica：一个藏着浪漫的名字

名字本身值得单独说一段，因为它几乎就是产品哲学的浓缩。

Multica 取自 Multiplexed Information and Computing Agent（多路复用的信息与计算智能体）。这是对 1964 年那个操作系统 Multics 的致敬。

Multics 的全称是 Multiplexed Information and Computing Service，由 MIT、贝尔实验室等联合开发，是最早的分时操作系统之一——它最了不起的贡献，是让多个用户分时共享同一台昂贵的大型机，每个人用起来都像独占整台机器。Multics 这个项目本身最终没做成，但它启发了 Ken Thompson 和 Dennis Ritchie 做出了一个更精简的版本 Unix（Unix 这个名字本身就是把 Multi 换成 Uni 的双关戏谑），而 Unix 后来成了 Linux、macOS、iOS、Android 几乎所有主流操作系统的鼻祖。

Multica 把 Multics 里的 Service 换成了 Agent，寓意「从多路复用的计算服务，进化为多路复用的智能体协作」。

这个类比之所以精准，是因为它点中了软件开发几十年来的一个根本约束：一直是「单线程」的——一个工程师，同一时间只能专注一件事，一次上下文切换。AI Agent 改变了这个等式。Multica 想做的，就是把「分时共享」这件事重新带回来，只不过这次分时复用整个系统的「用户」，既有人，也有自主运行的 Agent。用张佳圆的话说，赌的就是这个「多路复用」：有了对的系统，两个工程师加一队 Agent，可以像二十个人那样动起来。

## 二、它到底是什么：把 Agent 当员工的四个核心概念

理解了来龙去脉，再看产品本身就清晰多了。Multica 的产品形态是一个看板式界面，长得像 Linear 或 Jira，但看板上的参与者同时包括人和 Agent。它的运行核心由四个概念撑起来。

第一个是 Agent Runtime（运行时）。这是 Multica 一个关键的、克制的选择——它不自己再造一个 Agent，而是接入你已经在用的 Agent。你在本地装好的 Claude Code、Codex、Cursor Agent、GitHub Copilot CLI、OpenClaw、OpenCode、Hermes、Gemini、Pi、Kimi、Kiro CLI 等等，把运行环境注册到平台，就成了一个 Runtime。一个 Runtime 就是一台能执行任务的算力环境，可以是你本地机器（通过后台 daemon），也可以是云端实例。团队内部就注册了好几台机器——个人电脑、Mac mini 等，每台机器上的 Agent 都作为算力被调度。这一步从架构上把 Multica 定位成了「编排层」而非「Agent 本体」。

第二个是 Agent Profile（角色）。基于 Runtime 创建的具体角色，相当于一个「AI 员工」。每个 Agent 有自己的名字、记忆、Skills，可以定义角色和 Instructions，还能自我迭代。它会出现在看板上、在评论区发言、在任务里被分配。团队甚至开始探索更细的分工——专门做开发的、专门做市场推广（GTM）的。

第三个是 Issue（任务单元）。这是协作的最小载体：创建 Issue，分配给某个 Agent 或某个人，执行，完成后通知相关方 review。关键在于，Issue 里可以「@」其他成员，无论对方是人还是 Agent，从而实现真正的多方协作。一个 5 到 10 人团队的典型流程是这样的：周会定目标，Agent 批量创建 Issue，一部分直接派给 Agent、一部分派给人，Agent 自动执行，完成后「@」相关人 review，人在同一个 Issue 里直接给反馈，Agent 根据反馈迭代。整个过程，所有 context 都沉在一个地方，不用在飞书、终端、项目管理工具之间来回横跳。

第四个是 Inbox（收件箱）。它聚合所有需要你个人关注的事项，相当于每个人的「邮件收件箱」。人每天只需要盯 Inbox，点进去就有完整上下文。

在这四个核心之上，Multica 还叠了几个能力。Squads（小队）把若干 Agent 和人编成一个由 leader Agent 带队的小队，你把活派给整个小队，leader 负责分给最合适的成员，这样团队变大时路由依然稳定——你「@前端团队」而不是「@张三或李四或王五」。Autopilots（自动驾驶）给 Agent 排定时任务，用 cron、webhook 或手动触发，自动建 Issue 并派给 Agent，让日报、周报、定期审计自己跑起来。Reusable Skills（可复用技能）让每一次解决方案都能沉淀成全队可复用的技能，部署、迁移、code review 这些经验会随时间复利积累。

技术栈上，Multica 是一套相当现代的组合：前端 Next.js 16（App Router），后端用 Go（Chi 路由加 sqlc 加 gorilla/websocket），数据库是带 pgvector 的 PostgreSQL 17，本地一个 Agent Daemon 负责真正执行各家 CLI。它支持 Homebrew 一键安装、脚本安装，也支持加 `--with-server` 参数拉起完整的自托管服务（需要 Docker），甚至还有一个放在 `apps/mobile/` 里的 iOS 移动端。

把 Agent 当员工最直观的画面，是张佳圆描述的「团队的一天」：清晨，Research Agent 准点发出一份过去 24 小时的 AI 行业日报，数据分析 Agent 产出昨日数据报告，发现核心指标异常会自动建一个修复 Issue 派给决策人；白天，GitHub Agent 持续 review 外部 Issue 和 PR，bug 自动分析根因，feature 对齐 roadmap；代码产出后，Code Review Agent 先初审再交人终审；傍晚 6 点，Deploy Agent 定时触发，自动测试、打包、发布、更新 changelog。人在做的，只剩下最终决策——排优先级、把关质量、对需要拍板的事拍板。

## 三、横向看：同一个赛道里，大家都长成了什么样

「把 AI 编程 Agent 当队友、用看板管起来」这件事，Multica 不是唯一在做的。2026 年，这条赛道已经相当拥挤，但仔细看，每个玩家活成的样子差异很大。我们挑几个最具代表性的，逐一对照。

### Vibe Kanban：同样的看板，不同的野心

如果只看截图，Vibe Kanban 和 Multica 像极了——都是看板、都能给多个 Agent 派活、都做 Diff Review。但骨子里是两条路。

Vibe Kanban 的核心是一个 planning 加 review 的工作流，目标是让一个开发者更高效地调度和审阅多个编码 Agent。它的设计很「工程师本位」：每个 Issue 直接关联一个 Agent 工作区（Workspace），每个工作区给 Agent 分配一个独立的 Git Worktree、一个终端、一个开发服务器，Agent 在隔离环境里干活，产出直接体现为代码变更。它强调本地运行、不把代码发到外部服务器，用独立 Worktree 隔离保证代码库整洁。截至相关统计，它在 GitHub 上有两万多颗 Star。

Multica 的野心更大一层。它要做的不是「让一个人更高效地用多个 Agent」，而是「让一个团队（人加 Agent）在同一个空间里协作」。差别体现在那些 Vibe Kanban 没怎么碰的地方：Agent 有 Profile 和记忆、能主动建 Issue 和发评论、能被「@」、能组成 Squad、能跑 Autopilot。一句话概括：Vibe Kanban 是「给开发者的多 Agent 编排器」，Multica 是「给团队的人机协作操作系统」。前者优化的是个人产能，后者优化的是组织协作。

### Conductor / Claude Squad 这类编排器：更轻，更偏工具

以 Conductor、Claude Squad、dmux、amux 为代表的一批工具，本质是多 Agent 编排器。它们解决的是「我想同时跑好几个 Agent，怎么并行、怎么管理 worktree、怎么 review 合并」这一类很具体的工程问题。它们通常更轻量、更贴近终端、更适合单个重度用户。

和 Multica 比，这一类的共同点是「没有团队这一层」。它们默认使用者是一个人（顶多是几个各管各的工程师），没有把「Agent 是团队资产、要在人和人之间共享和协作」这件事做进产品骨架里。所以它们更像是「升级版的终端复用工具」，而 Multica 更像是「Linear 的人机协作版」。两者并非你死我活，很多团队完全可能在个人层面用 Conductor，在团队层面用 Multica。

### Devin：重资产的「AI 软件工程师」，完全另一条路

Devin（来自 Cognition Labs）是个值得放进来对照、但定位截然不同的对象。Devin 把自己定位成「AI 软件工程师」——它自己就是 Agent，你直接雇它干活，它端到端地完成任务。

这和 Multica 几乎是镜像关系：Devin 是「Agent 本体」，卖的是一个会写代码的 AI；Multica 是「编排层」，自己不产 Agent，而是把你现有的各家 Agent 组织起来。Devin 是闭源的、商业化的、重资产的 SaaS；Multica 是开源的、自托管的、厂商中立的基础设施。有意思的是，理论上 Devin 这样的 Agent 完全可以作为一个 Runtime 接入 Multica——它们处在价值链的不同位置，与其说是竞争，不如说一个可能是另一个的「员工」。

### Paperclip 这类「AI 公司」：更激进，但更像游戏

还有一批更激进的项目，比如把多个 AI 代理组织成一个有「组织架构、目标对齐、预算控制、治理机制、审计日志」的虚拟公司。这类项目的架构设计很前瞻——任务上下文继承、心跳调度、组织化委托，都是别人薄弱的环节。

但它们的问题也很典型，而且恰好能反衬出 Multica 的克制。其一，高度依赖外部 Agent，自己不产 Agent，如果你手上没有稳定支持「心跳」机制的 Agent，体验会大打折扣。其二，很多用户反映「概念很先进，但目前更像一个模拟公司的游戏」，离真实生产还有距离。其三，Token 成本风险，Agent 自主跑起来，上下文膨胀加频繁心跳，很容易导致 Token 爆炸，费用还得人盯着。

把这些放在一起看，Multica 的生态位就清楚了：它比 Conductor 这类编排器重（有完整的团队协作层），比 Paperclip 这类 AI 公司轻、也更落地（不强求 Agent 完全自治，始终把人留在最终决策的环节），和 Vibe Kanban 像但更团队化，和 Devin 则处在价值链的不同层。它找到了一个还算空旷的位置：面向「人加 Agent 混合团队」的、厂商中立的、开源自托管的协作基础设施。

下面这张表大致勾勒了各家的差异（仅作认知参照，具体数据会随时间变化）：

| 对象 | 本质定位 | 是否自产 Agent | 团队协作层 | 开源/自托管 | 一句话画像 |
|------|---------|:---:|:---:|:---:|------|
| Multica | 托管 Agent 协作平台 | 否（接入现有） | 强（Issue/Squad/Inbox） | 是 | 看板上一半是人一半是 Agent |
| Vibe Kanban | 多 Agent 编排器 | 否 | 弱（偏个人） | 是 | 给开发者的并行 Agent 看板 |
| Conductor/Claude Squad | 轻量编排器 | 否 | 几乎无 | 多为是 | 升级版终端复用工具 |
| Devin | AI 软件工程师 | 是（本体） | 无 | 否 | 直接雇一个会写代码的 AI |
| Paperclip 类 | 虚拟 AI 公司 | 否 | 强（治理/预算） | 是 | 前瞻但偏「模拟公司游戏」 |

## 四、用户怎么看：被夸的和被吐槽的

从公开讨论看，Multica 收获的正面评价集中在几点。一是理念清晰、踩点准——「把 Agent 当队友而非工具」这个定位戳中了大量已经在重度使用编程 Agent、却苦于协作混乱的团队。二是厂商中立、不绑架，你用什么 Agent 就接什么，不用换工具链，降低了迁移成本。三是代码质量和技术架构被认可，Go 加 Next.js 加 pgvector 的组合现代而扎实，自托管能力对数据敏感的团队很友好。四是「自己用自己」的说服力，一个 4 人团队靠它把开发、日报、部署、code review 都半自动化，这种自产自用本身极具传播力。

需要客观看待的另一面也存在。作为一个 2026 年才爆发的新项目，它仍处在早期：商业化路径尚在摸索（原计划 5 月才开始尝试），企业级能力（如 SSO、更完善的审计日志）还在补齐。更重要的是，整个品类共有的成本与可控性问题它也躲不开——当一群 Agent 在看板上自主跑任务，Token 消耗、上下文膨胀、以及「Agent 自主决策到什么程度才安全」这些问题，是所有这条赛道玩家都要长期面对的。Multica 的做法是始终把人留在「最终决策」这个环节，这是一种务实的折中，但也意味着它解决的是「协作效率」而非「完全替代人」。

把时间轴和竞争格局叠在一起看，Multica 的位置就清晰了：它站在「编程 Agent 已经够好用」这个拐点上，选了一个比单纯做工具更难、但天花板更高的方向——做人和 Agent 共同的协作层。它今天的优势（看板即协作中枢、Agent 作为一等公民、技能复用），本质上来自团队三年踩坑攒下的产品手感和对时机的判断；它今天的风险（品类太新、成本可控性、巨头随时可能下场），也都写在这个选择里。能不能从「GitHub 上的明星项目」走到「团队真正离不开的基础设施」，取决于接下来这一年它能否在商业化、稳定性和企业级能力上补齐短板，并守住「先发加开源生态」这道护城河。
