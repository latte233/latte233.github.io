> 每天有数十亿次下载从 npmjs.com 流出，支撑着全球几乎所有的 JavaScript 项目。但很少有人认真想过：这个每天被无数开发者敲下 `npm install` 的工具，是怎么来的？它经历了什么？它今天面临什么挑战？信息截止 2026 年 5 月，建议当作认知起点，感兴趣的地方自己再挖。

## 一、发展历程：从一个程序员的懒人工具到互联网基础设施

### 在 npm 出现之前，前端是什么样的

要理解 npm 为什么重要，得先回到它诞生之前的世界。

2009 年，前端开发者想用 jQuery，得去 jQuery 官网下载一个 `.js` 文件，放进自己的项目目录。想用 Bootstrap，再去 Bootstrap 官网下载。想用 Underscore，再去 Underscore 官网下载。每个库都有自己的官网、自己的下载页面、自己的版本管理方式。项目依赖多了，这件事就变成了一场噩梦——你得记住每个库的官网在哪、当前用的是哪个版本、有没有新版本可以升级。

更麻烦的是，这些库之间还有依赖关系。A 库依赖 B 库的某个版本，但你项目里用的 B 库是另一个版本，两者不兼容。这种"依赖地狱"（Dependency Hell）在当时是前端开发者的日常噩梦。

与此同时，服务端的 JavaScript 正在经历一场革命。2009 年 5 月，Ryan Dahl 在 JSConf EU 上发布了 Node.js——一个让 JavaScript 能在服务器端运行的运行时。Node.js 的出现打开了一扇门：JavaScript 不再只是浏览器里的玩具，它可以写服务器、写命令行工具、写任何东西。

但 Node.js 有一个明显的缺失：它没有包管理器。

### Isaac Schlueter 和 npm 的诞生

Isaac Z. Schlueter，一个在旧金山湾区工作的程序员，是 Node.js 早期社区的活跃成员。他有一个典型的程序员特质：懒。不是那种消极的懒，而是"我不想重复做同一件蠢事"的懒。

他受够了手动下载、手动管理依赖的方式。2009 年底，他开始动手写一个工具，思路很简单：建一个中央仓库，所有人把自己的代码包上传到这里；需要用某个包的人，一条命令就能下载下来，依赖关系自动处理。

2010 年 1 月，npm 正式发布，全称 Node Package Manager。

最初的 npm 极其简陋。没有网站，没有搜索，就是一个命令行工具加一个用 CouchDB 搭的简单数据库。Schlueter 给 Ryan Dahl 发了邮件，两人一拍即合——Node.js 需要一个包管理器，npm 需要一个宿主生态。2011 年，Node.js 开始内置 npm，这个决定从根本上改变了 npm 的命运。

从此，每一个安装 Node.js 的开发者，都自动拥有了 npm。

### 随 Node.js 一起爆发（2011-2014）

Node.js 的崛起速度超出了所有人的预期。2011 年，LinkedIn 宣布将移动后端从 Ruby on Rails 迁移到 Node.js，服务器数量从 30 台降到 3 台，性能提升 20 倍。这个案例在开发者社区广泛传播，Node.js 开始从"有趣的实验"变成"严肃的生产工具"。

Node.js 火了，npm 也跟着火了。

开发者们开始把自己写的工具、库、框架发布到 npm。jQuery 作者 John Resig 把 jQuery 发布到 npm，这是一个重要的信号——连前端最重要的库都来了，npm 的地位就此确立。到 2013 年，npm 上的包数量突破 3 万，周下载量达到 1479 万次。

这个阶段的 npm 还很粗糙。依赖管理采用嵌套结构（v1/v2 时代）：A 依赖 B，B 就装在 A 的 `node_modules` 里；C 也依赖 B，B 就在 C 的 `node_modules` 里再装一遍。这导致了臭名昭著的"node_modules 黑洞"——一个中等规模的项目，`node_modules` 目录动辄几百 MB，里面同一个包可能被安装了十几次。

但开发者们忍了。因为 npm 解决的问题太真实了，哪怕它本身还有很多问题。

2014 年，Schlueter 从 Node.js 核心维护者的位置退下来（他之前接手了 Ryan Dahl 离开后的 Node.js 维护工作），专心成立了 npm, Inc. 公司，专门运营 npm 注册表。这是 npm 从个人项目走向商业化的关键一步。

### 第一次危机：Yarn 的出现（2016）

2016 年 10 月，Facebook 发布了 Yarn。

这不是一个普通的竞品发布，而是一次对 npm 的公开批评。Facebook 的工程师们在博客里列出了 npm 的三大问题：安装速度慢、依赖安装结果不确定（同样的 `package.json`，不同时间安装可能得到不同的依赖树）、没有离线模式。

Yarn 的解决方案很优雅：引入 `yarn.lock` 文件锁定依赖版本，确保每次安装结果完全一致；并行下载包，速度比 npm 快 2-3 倍；本地缓存，支持离线安装。

Yarn 发布后，开发者社区的反应是爆炸性的。在 npm 积累了多年怨气的开发者们，把 Yarn 当成救星。GitHub 上的 star 数在几周内突破 2 万，各大公司纷纷宣布迁移到 Yarn。

这次冲击对 npm 来说是一次重要的警醒。npm 团队开始加速改进：2017 年，npm v5 发布，引入了 `package-lock.json`（直接借鉴了 Yarn 的 lock 文件思路），大幅改进了安装速度和缓存机制。npm 承认了自己的问题，并开始追赶。

### left-pad 事件：一个 11 行代码引发的全球故障（2016）

2016 年 3 月，一件荒诞的事情发生了，深刻暴露了 npm 生态的脆弱性。

一个叫 Azer Koçulu 的开发者，因为与 Kik Interactive 公司发生了包名争议（对方想要他的 `kik` 包名），一怒之下把自己在 npm 上发布的所有包全部删除了。其中有一个叫 `left-pad` 的包，功能极其简单——在字符串左边填充字符，代码只有 11 行。

但这个 11 行的包，被 Babel、React 等数千个项目直接或间接依赖。Koçulu 一删，全球无数项目的 `npm install` 开始报错，CI/CD 流水线大面积崩溃。Facebook、PayPal、Netflix、Spotify 的构建系统都受到了影响。

这个事件在开发者社区引发了深刻的讨论：我们真的应该依赖一个 11 行代码的包吗？npm 生态是不是过度碎片化了？一个人的情绪化决定，为什么能影响全球的软件构建？

npm 公司随后修改了政策：包一旦发布超过 24 小时，就不能被删除（除非有安全问题）。但这个事件留下的心理阴影，让很多开发者开始重新审视对 npm 生态的依赖方式。

### npm, Inc. 的内部动荡（2018-2019）

2018 年到 2019 年，npm 公司经历了一段混乱期。

先是 CEO 被指控性骚扰，随后辞职。接着是一系列管理层变动，多名核心工程师离职。公司内部文化问题被前员工公开披露，包括不透明的决策机制和对员工的不公平对待。

这段时间，npm 的产品迭代明显放缓，社区对 npm 公司的信任度下降。部分开发者和社区成员，因为对 npm 公司不满，开始推动创建替代方案——这直接催生了后来 pnpm 的快速发展。

### GitHub 收购：npm 找到了新家（2020）

2020 年 3 月，GitHub 宣布收购 npm, Inc.，收购金额未披露。

这个消息在开发者社区引发了复杂的反应。一方面，GitHub（背后是微软）的资源和信誉，能给 npm 提供更稳定的运营保障；另一方面，JavaScript 生态最重要的两个基础设施——代码托管（GitHub）和包管理（npm）——都落入了同一家公司手中，这让一些人感到不安。

GitHub CEO Nat Friedman 在公告中承诺：npm 注册表将继续免费、公开，不会关闭服务；GitHub 将投入资源改善 npm 的基础设施和安全性。

收购后，npm 的基础设施确实得到了明显改善。注册表的稳定性提升，安全审计功能加强，与 GitHub Actions 的集成也越来越深。2021 年，npm 推出了双因素认证强制要求，开始认真对待供应链安全问题。

### npm 的技术演进：从嵌套到扁平，再到内容寻址

npm 的依赖管理机制，经历了三个重要阶段的演进，每一次都是对上一代问题的回应。

**v1/v2 时代：嵌套结构。** 依赖关系完全按树形嵌套存放。A 依赖 B，B 就在 `A/node_modules/B`；C 也依赖 B，B 就在 `C/node_modules/B`。逻辑清晰，但导致严重的重复安装问题——同一个包可能在 `node_modules` 里出现几十次，目录层级深到 Windows 文件系统都处理不了。

**v3 时代：扁平化（hoist 机制）。** npm v3 引入了"提升"机制：把所有依赖尽量提升到顶层 `node_modules`，只有版本冲突时才保留嵌套。这解决了重复安装问题，但带来了新问题：幽灵依赖（你的代码可以引用 `package.json` 里没有声明的包，因为它被提升到了顶层）和依赖分身（同一个包的不同版本仍然会被重复安装）。

**v5 时代：lock 文件 + 内容寻址缓存。** npm v5 引入 `package-lock.json`，精确锁定每个依赖的版本和来源，解决了安装不确定性问题。同时改进了缓存机制，采用内容寻址存储（Content-Addressable Storage），相同内容的包只存一份，大幅减少磁盘占用。

**v7/v8/v9 时代：workspaces 和持续优化。** npm v7 引入了 workspaces 支持，让 monorepo 管理成为可能。v8、v9 持续优化性能，修复安全漏洞，缩小与 Yarn、pnpm 的差距。

### 2025 年的供应链安全危机

2025 年 9 月 8 日，npm 生态遭遇了历史上最大规模的供应链攻击之一。

攻击者通过针对知名包维护者 Josh Junon（GitHub 用户名 Qix-）的钓鱼攻击，获取了其 npm 账号控制权，随后向 18 个高人气包注入了恶意代码。这 18 个包的周下载量合计超过 26 亿次，受影响范围极广。恶意代码的目标是劫持加密货币钱包交易。

恶意版本在线约 2.5 小时后被发现并下架，但这次事件的影响远超技术层面。美国网络安全和基础设施安全局（CISA）专门发布了安全警报，要求相关组织检查依赖。这次攻击再次证明：npm 注册表作为全球最大的软件包仓库，其安全性已经是互联网基础设施级别的问题，任何一个环节的失守都可能引发连锁反应。

到 2025 年底，npm 注册表上的包数量已超过 300 万，每周下载量超过 500 亿次。这个数字既是 npm 成功的证明，也是它安全压力的来源。

## 二、竞品对比：包管理器的"战国时代"

npm 的竞争格局属于典型的"场景 C"——竞品充分，且各有鲜明特色。主要竞争者是 Yarn、pnpm 和 Bun，三者都在 npm 的某个痛点上做了针对性的改进，但也各自带来了新的取舍。

### Yarn：第一个挑战者，如今有些尴尬

Yarn 是 Facebook（现 Meta）在 2016 年发布的，它的出现直接逼着 npm 做了大量改进。但 Yarn 自己的发展轨迹，却走出了一条有些曲折的路。

Yarn 1.x（Classic）是大多数人熟悉的版本：`yarn.lock`、并行安装、离线缓存。它在 2016-2019 年间是很多团队的首选，尤其是 React 生态的项目（毕竟 Facebook 出品）。

但 Yarn 2.x（Berry）的发布，让很多用户感到困惑甚至愤怒。Yarn 2 引入了"Plug'n'Play"（PnP）机制，彻底抛弃了 `node_modules` 目录，改用一个 `.pnp.cjs` 文件来管理依赖解析。理论上，PnP 能彻底解决幽灵依赖问题，安装速度更快，磁盘占用更少。

但实际上，PnP 的兼容性问题让很多开发者头疼。大量工具（webpack、Jest、TypeScript 等）需要额外配置才能与 PnP 兼容，迁移成本很高。Yarn 2 还改变了很多 CLI 命令的行为，与 Yarn 1 不完全兼容，升级体验很差。

社区的反应是：很多人选择留在 Yarn 1，或者直接迁移到 pnpm，而不是升级到 Yarn 2。Yarn 2/3/4 的用户增长明显慢于 pnpm。

到 2025 年，Yarn 的市场份额约为 15-20%，主要集中在已有 Yarn 1 项目的团队，以及 Meta 系的项目。新项目选择 Yarn 的比例在下降。

**用户真实评价：** 老用户觉得 Yarn 1 够用，不想升级；新用户觉得 Yarn 2 的 PnP 太激进，学习成本高；很多人的感受是"Yarn 曾经很好，但现在不知道该用哪个版本"。

### pnpm：技术上最优雅的解决方案

pnpm（Performant npm）由 Zoltan Kochan 在 2017 年发布，是目前增长最快的包管理器。

pnpm 的核心创新是"内容寻址存储 + 硬链接"机制。所有包只在全局存储一份（按内容哈希存储），项目的 `node_modules` 里的文件都是指向全局存储的硬链接。这意味着：

- 同一个包的同一个版本，无论被多少个项目使用，磁盘上只存一份。在有多个项目的开发机上，磁盘节省效果极其显著，通常能节省 60-70% 的空间。
- 安装速度快，因为大量包已经在全局缓存里，只需要创建硬链接，不需要重新下载和解压。
- 严格的依赖隔离：pnpm 不做 hoist，`node_modules` 里只有 `package.json` 声明的直接依赖，彻底消灭了幽灵依赖问题。

pnpm 的 monorepo 支持也非常出色，`pnpm workspaces` 的使用体验被普遍认为优于 npm workspaces 和 Yarn workspaces。

**用户真实评价：** 开发者对 pnpm 的评价普遍很高，尤其是在 monorepo 场景下。常见的评价是"用了就回不去了"。主要槽点是：严格的依赖隔离有时会导致一些依赖配置不规范的包出现问题，需要手动配置 `shamefully-hoist` 来兼容；对于不熟悉硬链接机制的开发者，遇到问题时排查起来比较困难。

到 2025 年，pnpm 的市场份额约为 10-15%，但在新项目和技术型团队中的采用率更高。Vue、Vite、Nuxt 等知名开源项目都已迁移到 pnpm。

### Bun：速度至上的新挑战者

Bun 是 2022 年发布的，由 Jarred Sumner 用 Zig 语言编写，定位是"全能 JavaScript 运行时"——它不只是包管理器，还是 JavaScript 运行时、打包工具、测试框架。

Bun 的包管理器部分，最大的卖点是速度。在基准测试中，Bun 的安装速度通常比 npm 快 10-30 倍，比 pnpm 快 3-5 倍。这个速度差异在 CI/CD 环境中尤其明显，能显著缩短构建时间。

但 Bun 的问题也很明显：兼容性。Bun 的运行时并不完全兼容 Node.js，一些依赖 Node.js 特定 API 的包可能无法正常工作。作为包管理器单独使用时，兼容性问题相对少一些，但仍然存在边缘情况。

**用户真实评价：** 开发者对 Bun 的速度印象深刻，但对其稳定性和兼容性有顾虑。常见的使用模式是：在个人项目或新项目中尝试 Bun，在生产环境的大型项目中仍然使用 npm 或 pnpm。

到 2025 年，Bun 的市场份额约为 3-5%，但社区关注度很高，增长势头明显。

### 竞争格局总结

| 维度 | npm | Yarn | pnpm | Bun |
|------|-----|------|------|-----|
| 安装速度 | 中等（v9 后有改善） | 中等（PnP 模式快） | 快 | 极快 |
| 磁盘占用 | 较高 | 中等 | 最低 | 较高 |
| 依赖隔离 | 弱（幽灵依赖问题） | 弱（Classic）/ 强（PnP） | 强 | 中等 |
| Monorepo 支持 | 一般 | 良好 | 优秀 | 一般 |
| 兼容性 | 最好（事实标准） | 良好 | 良好 | 一般 |
| 学习成本 | 最低 | 中等 | 低 | 中等 |
| 市场份额（2025） | ~65% | ~15% | ~12% | ~4% |

npm 的最大优势，是它的"默认性"。它随 Node.js 一起安装，是所有教程、文档、CI 模板的默认选项。这种惯性极其强大，即使技术上不是最优解，也能维持最大的市场份额。

## 三、综合判断：npm 的过去、现在与未来

### npm 是什么？

在技术层面，npm 是一个包管理器。但如果只把它理解成一个工具，就低估了它的意义。

npm 是 JavaScript 生态的基础设施。它建立了一套"发布-发现-安装"的标准流程，让全球数百万开发者能够共享代码。没有 npm，就没有今天的 Node.js 生态，也就没有今天的前端工程化。React、Vue、Angular、webpack、Babel——这些塑造了现代前端开发的工具，都是通过 npm 传播和使用的。

从这个角度看，npm 的成功不只是一个产品的成功，而是一种协作模式的成功：开放的注册表 + 标准化的包格式 + 简单的命令行工具，三者结合，创造了一个自我强化的生态系统。

### npm 的核心矛盾

npm 今天面临的最大矛盾，是"规模"与"安全"之间的张力。

300 万个包、每周 500 亿次下载，这个规模让 npm 成为了互联网最重要的软件分发渠道之一。但同样的规模，也让它成为了攻击者的高价值目标。2025 年 9 月的供应链攻击，只是这个矛盾的最新一次爆发。

npm 注册表的开放性是它成功的根基——任何人都可以发布包，没有审核门槛。但这种开放性，也意味着恶意包可以轻易混入。npm 团队在安全方面做了大量工作（双因素认证、自动安全审计、恶意包检测），但在一个如此庞大的生态里，安全永远是一场猫鼠游戏。

### npm 的竞争地位：护城河与挑战

npm 的护城河，本质上是网络效应。300 万个包、数千万开发者、无数教程和文档——这些都指向 npm。切换包管理器的成本，不只是技术成本，还有认知成本和生态成本。

但这条护城河并不是无懈可击的。pnpm 的崛起证明，如果技术优势足够明显，开发者是愿意切换的。pnpm 在 monorepo 场景下的优势，已经让很多大型项目完成了迁移。

npm 的应对策略，是"够用就好"。它不需要在每个技术维度上都是最优解，只需要在大多数场景下足够好，同时保持最广泛的兼容性。npm v9 的性能已经比 v6 时代好了很多，"npm 很慢"的刻板印象在单项目场景下已经不太准确了。

### 未来走向

npm 的未来，大概率是"稳定的基础设施"而不是"激进的创新者"。

在注册表层面，npm 会继续是 JavaScript 生态的事实标准。没有任何竞争者有能力或意愿建立一个能替代 npmjs.com 的注册表——这需要巨大的存储、带宽和运营成本，以及多年积累的包生态。

在 CLI 层面，npm 会继续被 pnpm、Yarn、Bun 蚕食市场份额，尤其是在技术型团队和新项目中。但 npm 的"默认"地位，会让它在相当长的时间内保持最大的用户基数。

在安全层面，供应链安全会成为 npm 未来几年最重要的战场。GitHub 的资源和安全能力，会在这里发挥关键作用。可以预期的方向包括：更严格的包发布审核、更完善的依赖漏洞追踪、与 GitHub 代码仓库的更深度集成（让包的来源可追溯）。

有一个值得关注的趋势：Deno 和 Bun 都在尝试绕过 npm 注册表，直接从 URL 或 JSR（JavaScript Registry）导入模块。这代表了一种不同的依赖管理哲学——去中心化、URL 优先。这个方向短期内不会撼动 npm 的地位，但长期来看，它可能是 npm 模式最根本的挑战者。

### 一个更大的问题

npm 的故事，其实是一个关于"开放生态的代价"的故事。

npm 的成功，建立在极低的发布门槛上：任何人都可以发布包，没有审核，没有质量标准。这种开放性催生了繁荣的生态，但也催生了 left-pad 这样的荒诞事件，以及 2025 年的供应链攻击。

我们依赖的软件，越来越多地建立在无数陌生人写的、我们从未审查过的代码上。npm 让这种依赖变得极其便利，但便利的背后，是我们集体承担的安全风险。

这不是 npm 独有的问题，而是整个开源生态的问题。但 npm 因为规模最大，这个问题在它身上体现得最为突出。

如何在开放性和安全性之间找到平衡，是 npm 未来最重要的命题，也是整个软件行业需要认真面对的问题。
