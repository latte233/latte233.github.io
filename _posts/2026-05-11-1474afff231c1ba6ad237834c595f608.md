> 这篇东西写了很久，因为我越研究越觉得不对劲。SDD 听起来太好了——先写规格再写代码，AI 照着规格来，就不会乱来了。但真的用了之后，你会发现事情没这么简单。信息截止 2026 年 5 月，建议当作起点，感兴趣的地方自己再挖。

## 一、从 Vibe Coding 的宿醉说起

### Karpathy 点了一把火

2025 年 2 月，Andrej Karpathy 发了条推，大意是：他现在写代码不看代码了，把想法告诉 AI，AI 生成什么就用什么，不对就重来，全凭感觉。他把这叫 **Vibe Coding**。

这条推炸了。说白了，很多人早就在这么干了，只是不好意思说。Karpathy 给了它一个名字，突然就变得名正言顺了。

接下来几个月，Cursor、Windsurf、GitHub Copilot 用户量暴涨，“10 倍速开发”的故事到处传。更有意思的是，非程序员也开始写代码了——产品经理、设计师、没写过一行代码的创业者，都能 Vibe 出自己的 MVP。

但蜜月期没撑多久。

### 宿醉来了

2025 年中开始，数据开始打脸。

Veracode 的调查说 AI 生成的代码里 **45% 有安全漏洞**。CodeRabbit 分析了 470 个开源 PR，发现 AI 协作的代码比人写的“主要问题”多 1.7 倍，配置错误多 75%，安全漏洞多 2.74 倍。更有意思的一个研究：**熟练开发者用 AI 编码工具，生产力反而掉了 19%**——但他们自己预测会提升 24%。

GitClear 对 2.11 亿行代码的研究还发现另一个问题：AI 工具普及后，复制粘贴率涨了 48%，真正的重构活动掉了 60%。代码在变多，质量在变差，可维护性在恶化。

我管这叫“Vibe Coding 宿醉”。根子不难理解——Vibe Coding 的核心假设是“AI 能读懂你的意图”，但 LLM 本质上是模式补全机器，它擅长生成“看起来对”的代码，而不是“真正对”的代码。

### SDD 登场：旧酒装新瓶

Vibe Coding 的问题暴露了，一个新的词开始流传：**Spec-Driven Development（SDD，规范驱动开发）**。

SDD 的核心思想不新鲜。它脱胎于软件工程里的“契约优先”思想，以及 2000 年代的行为驱动开发（BDD）。BDD 的理念是：写代码之前，先用自然语言描述系统行为（Given/When/Then），让这些描述成为开发、测试和业务之间的共同语言。

SDD 把这个搬到了 AI 时代，做了一个关键重新定位：**Spec 不只是写给人看的文档，它是写给 AI 看的“执行指令”。**

2025 年下半年，Thoughtworks 在技术雷达里把 SDD 列为重点关注实践。技术总监 Liu Shangqi 说了一句被广泛引用的话："Spec-driven development may not have the visibility of a term like vibe coding, but it's nevertheless one of the most important practices to emerge in 2025."

### 工具生态迅速成型

2025 到 2026 年初，几个玩家陆续登场：

**GitHub Spec Kit**（2025 年中）：GitHub 官方的 SDD 工具包，基于 Markdown，通过结构化 spec 引导 AI 生成代码。设计哲学是“严格阶段门”——每个阶段必须完成才进下一步。

**Amazon Kiro**（2025 年下半年）：AWS 的专属 IDE，深度集成 SDD 工作流。Spec 直接嵌在 IDE 里，边写代码边看规范。但问题也明显：锁死在 Kiro IDE + Claude 模型，不支持其他工具链。

**OpenSpec**（2025 年底，Fission-AI 开源）：定位“通用开源框架”，支持 20+ 主流 AI 编程助手。三条命令上手：`/opsx:propose`（提出变更，生成四类文档）、`/opsx:apply`（逐任务执行）、`/opsx:archive`（归档变更）。

**Tessl**（2026 年初）：最激进的一个——spec 是唯一维护的产物，代码完全由 spec 生成，不需要人维护代码库。SDD 的极端形态，还处于早期。

2026 年 2 月，arxiv 出了一篇《Spec-Driven Development: From Code to Contract in the Age of AI Coding》，系统梳理了 SDD 的理论基础。学术界也开始关注了。

到 2026 年，SDD 从工程圈小众讨论变成了 AI 编程主流范式之一。阿里、腾讯等大厂开始发内部实践经验，“规范驱动”成了 AI 原生研发的标配词。

## 二、SDD 确实解决了一些真问题

在说问题之前，先承认它解决了什么——不然没法理解为什么它能在这么短时间内被广泛认可。

**上下文锚定**：Vibe Coding 最头疼的工程问题是“上下文腐烂”——聊着聊着，AI 就忘了之前的设计决策，代码质量直线下降。SDD 把决策固化在 Markdown 文件里，纳入版本控制。任何时候把 spec 交给 AI，它就能重建完整上下文。

**幻觉抑制**：结构化的 spec 给 AI 提供了精确输入，把推理路径限制在更小范围内，减少了“自由发挥”导致的幻觉。OpenSpec 的“小步快跑”模式（每次只处理一个原子任务）进一步强化了这个效果。

**审查范式变了**：Vibe Coding 时代，AI 生成的上万行代码你根本没法逐行 Review。SDD 把“代码审查”变成了“规范审查”——代码生成之前先审查 spec 对不对，成本低，介入早。

**知识能留下来了**：Vibe Coding 里项目知识存在聊天历史中，极易丢。SDD 把知识沉淀在 `specs/` 目录，换 AI 模型也不怕，新模型读规范就能快速理解项目全貌。

这些都是真的，也是 SDD 快速获得认可的原因。但认可一个范式的价值，和承认它已经成熟，是两回事。

## 三、五个还没解决的问题

SDD 在 2025-2026 的快速崛起，掩盖了一个事实：这范式本身还有大量根本性问题没解决。Thoughtworks 的 Liu Shangqi 在肯定 SDD 价值的同时，也直接说了："Spec drift and hallucination are inherently difficult to avoid."

我梳理了五个最核心的。

### Spec Drift：规范和代码的同步是个伪命题

SDD 的核心假设是“规范是单一事实来源”。但实际工程中，这个假设很快就会碎。

代码在变，规范却很难同步更新。典型场景：开发者按 spec 生成了代码，调试时直接改了代码（因为比改 spec 再重新生成快多了），但没回头更新 spec。几周后，spec 和代码之间出现偏差。这种偏差叫 **Spec Drift（规范漂移）**。

Spec Drift 危险在它是隐性的。代码能跑，测试能过，但规范已经不再准确描述系统实际行为了。下一个开发者（或下一个 AI 会话）拿着过时 spec 来生成新代码，就会基于错误前提做实现。

更深层的问题：**谁来维护规范的准确性？** 传统软件工程里文档腐烂是个老问题，解决方案是“代码即文档”——让代码本身成为最准确的描述。SDD 试图反转这个逻辑，让规范成为最准确的描述，但它没解决“谁保证规范和代码同步”的问题。

目前 OpenSpec、Spec Kit 这些工具都没有自动检测 Spec Drift 的机制。OpenSpec 的 `/opsx:archive` 能归档变更，但依赖开发者主动执行。靠人类自律的方案，工程实践中往往不靠谱。

### Spec 质量的冷启动：谁来写第一份好规范？

SDD 的工作流通常是这样：人提需求 → AI 生成 spec → 人审查 spec → AI 按 spec 生成代码。前提是“人能审查 spec 质量”。

但这里有个循环依赖：**要审查一份好 spec，得对系统有深入理解；但如果对系统有深入理解，可能根本不需要 AI 帮你写 spec。**

对有经验的开发者，SDD 价值很明显——他们能快速识别 AI 生成的 spec 里的问题，补充边界条件，修正错误假设。但对经验不足的开发者（或者非技术背景的人，他们恰恰是 Vibe Coding 的主要受益者），判断一份 spec 够不够好很难。他们会批准一份看起来合理但遗漏了关键约束的 spec，然后得到一份看起来正确但有严重缺陷的代码。

多代理系统里这问题更严重。多个 AI Agent 协作时，一个 Agent 生成的 spec 被另一个 Agent 直接使用，中间没人审查。第一个 Agent 的 spec 有问题，错误会被放大传播，形成“幻觉循环”——Agent A 的错误输出被 Agent B 误判为正确，相互强化，最终产出看起来自洽但实际完全错误的系统。

目前没有工具提供“spec 质量评估”的自动化机制。判断一份 spec 够不够完整、够不够准确、够不够无歧义，还是得靠人。

### 验证缺口：规范说了，AI 不一定做到

即使有了高质量的 spec，还有一个根本性的技术问题：**LLM 是非确定性的**。同样的 spec，不同模型、不同时间、不同温度参数，可能生成完全不同的代码。

这意味着“spec 通过审查”不等于“生成的代码符合 spec”。你得有个独立的验证机制确认代码确实实现了 spec 描述的行为。

传统方案是测试。但 SDD 语境下这引入了新问题：**测试本身也是 AI 生成的**。如果 AI 生成的测试和代码都基于同一份 spec，那测试可能只是在验证“AI 对 spec 的理解是否自洽”，而不是在验证“代码是否真正正确”。

具体来说：如果 spec 里有个模糊表述，AI 生成代码时做了某种解释，生成测试时也会做同样的解释。测试会过，但代码行为可能不是你真正想要的。

arxiv 那篇论文明确提了这个问题，但目前没有系统性解决方案。Thoughtworks 的建议是“高度确定性的 CI/CD 实践”，但这更像一个方向，不是具体方法。

### 工具碎片化：SDD 的“标准战争”

2026 年的 SDD 工具生态，碎片化得令人担忧。

OpenSpec、GitHub Spec Kit、Amazon Kiro、Tessl——每个都有自己的 spec 格式、工作流、命令体系，互相不兼容。一个团队用 OpenSpec 积累了大量 spec 文件，想切到 Kiro，基本得从头来。

这带来几个问题：

**生态锁定风险**：Kiro 绑了 AWS 和 Claude，Spec Kit 绑了 GitHub 生态。选一个工具，就得接受背后的生态绑定。对企业用户来说，这是重大战略风险。

**最佳实践没法共享**：不同工具的 spec 格式不同，一个团队积累的 spec 写作经验和模板，没法直接迁移。整个行业的学习成本被人为放大。

**“好 spec”没有标准**：目前没有行业标准定义什么是“好的 spec”。不同工具要求不同，不同团队质量参差不齐。SDD 的效果高度依赖团队具体实践，难以规模化。

arxiv 论文里明确说："there is no universal 'good spec' evaluation yet"。这不只是学术问题，直接关系到 SDD 能不能从个人实践变成团队级、组织级的工程能力。

### 认知负担的转移：SDD 并没有让开发变简单

SDD 的支持者常把它描述为“让 AI 编程更可靠”的方案。但一个经常被忽略的事实是：**SDD 没有降低总体认知负担，它只是把负担从“写代码”转移到了“写规范”**。

Vibe Coding 里，认知负担主要是“怎么描述需求让 AI 理解”和“怎么验证 AI 生成的代码”。SDD 里，认知负担变成了“怎么写出足够精确、足够完整、足够无歧义的规范”。

对有经验的开发者，这种转移是有价值的——写规范比写代码更接近“设计”，更能发挥判断力。但对经验不足的开发者，写一份好规范可能比直接写代码更难。

阿里技术团队在内部实践报告里坦承了这个问题：SDD 的“标准版”（Team Mode）需要完整的 Requirements → Interface → Implementation 链路，对团队工程能力要求相当高。他们提供了“Lite 版”（Vibe Mode）作为降级方案，但 Lite 版的效果和 Vibe Coding 差距不大。

更深层的问题是：**SDD 假设人能在编码之前把所有需求想清楚**。但软件开发的现实是，很多需求只有实现过程中才会变清晰。“先写规范再写代码”的线性流程，和软件开发的探索性本质存在根本张力。

不是说 SDD 错了，而是它更适合需求相对明确的场景（功能迭代、接口实现），不适合需求模糊的探索性场景（新产品原型、技术验证）。但目前 SDD 的工具和方法论，往往没有清晰区分这两种场景。

## 四、工具对比：三条路线

### OpenSpec：通用但浅

OpenSpec 是目前最通用的 SDD 框架，MIT 协议，支持 20+ AI 工具。核心优势是“低侵入性”——三条命令上手，不用换工具链。

但通用性的代价是深度。因为要兼容所有工具，没法深度集成任何一个的特性。Spec 格式是纯 Markdown，没有机器可读的结构化约束，质量全靠写作者能力，工具本身没法提供质量保证。

社区反馈是：个人开发者和小团队用着不错，但大型团队里缺乏强制质量门禁，spec 质量参差不齐，最终退化成“有 spec 格式的 Vibe Coding”。

### GitHub Spec Kit：标准强但门槛高

设计哲学是“严格阶段门”——每个阶段必须完成才进下一步。团队协作和可追溯性出色，但学习曲线和使用摩擦也高。

另一个问题是主要针对新项目设计，对已有代码库（Brownfield Projects）支持不够。实际工程中，大多数开发是在已有系统上迭代，不是从零开始。

### Amazon Kiro：体验最好但锁得最死

三者中集成度最高，spec 工作流嵌在 IDE 里，体验最顺滑。但生态锁定最严重：Kiro IDE + Claude 模型 + AWS 生态。对已有工具链的团队，迁移成本极高。

用户反馈两极分化：喜欢的人觉得是目前体验最好的 SDD 工具，不喜欢的人觉得是精心设计的 AWS 锁定陷阱。

### Tessl：最激进的赌

Spec 是唯一维护的产物，代码完全由 spec 生成。SDD 的逻辑终点。

但目前太早期了。代码完全由 spec 生成时，性能优化、调试、遗留系统集成这些需要直接操作代码的场景怎么搞？没人知道。

| 维度 | OpenSpec | GitHub Spec Kit | Amazon Kiro | Tessl |
|------|----------|-----------------|-------------|-------|
| 工具绑定 | 无（支持 20+） | GitHub 生态 | Kiro IDE + Claude | 独立 |
| 适用场景 | 新旧项目均可 | 主要新项目 | 新项目 | 早期探索 |
| 学习曲线 | 极低 | 较高 | 中等 | 高 |
| 质量门禁 | 弱（靠自律） | 强（强制阶段门） | 中等 | 待定 |
| 开源 | MIT | MIT | 闭源 | 闭源 |

目前没有一个工具能同时做到“低摩擦”和“高质量保证”。这个矛盾是 SDD 工具生态的核心张力。

## 五、我的判断

SDD 的崛起是真实的，解决的问题也是真实的。AI 编程从“个人玩具”走向“团队工程”，某种形式的规范化是必然的。SDD 提供了一个合理的思考框架。

但把它描述成“AI 编程的终极方案”，过度乐观了。

它解决了上下文管理的问题，但没解决规范质量的问题。一份好 spec 需要深厚工程经验，而这恰恰是很多 AI 编程用户缺乏的。SDD 把认知负担从“写代码”转到了“写规范”，但没降低总体认知门槛。

它提供了框架，但没提供验证机制。规范和代码之间的一致性验证，目前全靠人工审查和测试。AI 生成代码的速度远超人工审查速度，这个瓶颈只会越来越明显。

它的工具生态还处于“战国时代”。没有统一标准，没有互操作性，没有“好 spec”的评估基准。效果高度依赖团队具体实践，难以规模化。

它与软件开发的探索性本质存在张力。“先写规范再写代码”更适合需求明确的场景，不适合需求模糊的探索性场景。但目前 SDD 方法论往往没区分这两种场景。

从历史角度看，SDD 很像 2000 年代的 BDD——理念正确，工具也在快速成熟，但大规模落地需要更长时间，需要工具链成熟、最佳实践沉淀、开发者认知转变。

我觉得最可能的走向：SDD 不会取代 Vibe Coding，而是和它共存，形成“分层”生态——探索性工作用 Vibe Coding，生产级工作用 SDD。工具会逐渐融合，最终形态可能是“智能 spec 辅助”——AI 不只按 spec 生成代码，还能主动检测 Spec Drift、评估 spec 质量、在 spec 和代码之间建立双向同步。

但这个未来还需要时间。2026 年的 SDD，是一个正确方向上的早期实践，不是一个已经解决了核心问题的成熟范式。
